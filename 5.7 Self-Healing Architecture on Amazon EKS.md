# Self-Healing Architecture on Amazon EKS

AWS-centric build guide, EKS at the core throughout. Matches `self-healing-eks-architecture.svg`: AWS Load Balancer Controller for ingress, kubelet for pod-level healing, an EKS Managed Node Group for node-level healing, CloudWatch Container Insights → EventBridge → Lambda for automated remediation, Amazon Bedrock for failure diagnosis, and AWS FIS's native EKS actions to prove it all works.

Estimated build time: 4–6 hours. Estimated cost: a few dollars/day for the cluster and node group while it's running — see Cleanup.

---

## Prerequisites

- AWS CLI, `kubectl`, `eksctl`, and `helm` installed and configured
- An AWS account with permissions for EKS, EC2, IAM, Lambda, EventBridge, CloudWatch, Bedrock, and FIS
- A region picked and used consistently (e.g., `us-east-1`) — check that Bedrock model access is enabled in that region for your account

---

## Phase 1: Stand up the EKS cluster

```bash
eksctl create cluster \
  --name selfheal-eks \
  --region <region> \
  --nodegroup-name selfheal-ng \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 2 \
  --nodes-max 5 \
  --managed
```
`--managed` is the key flag — it creates an **EKS Managed Node Group**, which is what gives you node-level self-healing for free (it's backed by an Auto Scaling Group under the hood).

`eksctl` also enables an IAM OIDC provider on the cluster by default, which you'll need in Phase 4 for Lambda to authenticate against the Kubernetes API (IRSA-style).

## Phase 2: Ingress via the AWS Load Balancer Controller

### 2.1 Install the controller
```bash
helm repo add eks https://aws.github.io/eks-charts
helm repo update
eksctl create iamserviceaccount \
  --cluster selfheal-eks --namespace kube-system --name aws-load-balancer-controller \
  --attach-policy-arn arn:aws:iam::aws:policy/ElasticLoadBalancingFullAccess \
  --approve
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=selfheal-eks \
  --set serviceAccount.create=false \
  --set serviceAccount.name=aws-load-balancer-controller
```
(Use a scoped-down IAM policy in production — the AWS Load Balancer Controller docs publish the minimal policy JSON.)

### 2.2 Deploy the app and an ALB-backed Ingress
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: selfheal-web
spec:
  replicas: 3
  selector: { matchLabels: { app: selfheal-web } }
  template:
    metadata: { labels: { app: selfheal-web } }
    spec:
      containers:
        - name: web
          image: nginxdemos/hello
          ports: [{ containerPort: 80 }]
          livenessProbe: { httpGet: { path: /, port: 80 }, initialDelaySeconds: 5, periodSeconds: 5 }
          readinessProbe: { httpGet: { path: /, port: 80 }, initialDelaySeconds: 3, periodSeconds: 5 }
---
apiVersion: v1
kind: Service
metadata: { name: selfheal-web, annotations: { alb.ingress.kubernetes.io/target-type: ip } }
spec:
  type: NodePort
  selector: { app: selfheal-web }
  ports: [{ port: 80, targetPort: 80 }]
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: selfheal-ingress
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/scheme: internet-facing
spec:
  rules:
    - http:
        paths:
          - path: /
            pathType: Prefix
            backend: { service: { name: selfheal-web, port: { number: 80 } } }
```
```bash
kubectl apply -f app.yaml
kubectl get ingress selfheal-ingress   # note the ALB hostname once provisioned
```

## Phase 3: The two self-healing tiers

**Tier 1 — kubelet (pod layer).** Kill a container's main process directly and watch it restart in place, same pod:
```bash
kubectl exec <pod> -- kill 1
kubectl get pod <pod> -w
```

**Tier 2 — EKS Managed Node Group (node layer).** Terminate the EC2 instance backing a node and watch the node group's ASG replace it, with pods rescheduling onto healthy nodes:
```bash
kubectl get nodes -o wide
aws ec2 terminate-instances --instance-ids <instance-id-behind-a-node>
kubectl get nodes -w
kubectl get pods -o wide -w
```
This is the direct EKS equivalent of an AWS Auto Scaling Group replacing a failed EC2 instance — because that's exactly what's running underneath the managed node group.

## Phase 4: Automated remediation — CloudWatch → EventBridge → Lambda → Bedrock

**Goal:** detect an application-level problem and have Lambda decide the right fix, using Bedrock to classify the failure before acting.

### 4.1 Enable CloudWatch Container Insights
```bash
aws eks create-addon --cluster-name selfheal-eks --addon-name amazon-cloudwatch-observability
```
This gives you container and pod-level metrics (including restart counts) in the `ContainerInsights` CloudWatch namespace with no extra agents to manage.

### 4.2 Create a CloudWatch Alarm
- CloudWatch Console → Alarms → Create alarm
- Namespace: `ContainerInsights`, metric: `pod_number_of_container_restarts` (or your app's custom error metric if you're emitting one), dimensions scoped to your cluster/namespace/pod
- Threshold: e.g., `>= 3` restarts over 5 minutes
- Name: `selfheal-restart-alarm`

### 4.3 Grant the Lambda access to the Kubernetes API (EKS access entries)
```bash
aws eks create-access-entry \
  --cluster-name selfheal-eks \
  --principal-arn <lambda-execution-role-arn> \
  --type STANDARD
aws eks associate-access-policy \
  --cluster-name selfheal-eks \
  --principal-arn <lambda-execution-role-arn> \
  --access-scope type=namespace,namespaces=default \
  --policy-arn arn:aws:eks::aws:cluster-access-policy/AmazonEKSEditPolicy
```
This is the EKS-native replacement for hand-editing the `aws-auth` ConfigMap — it lets the Lambda's IAM role authenticate to the cluster and act with RBAC scoped to the `default` namespace only.

### 4.4 Build the remediation Lambda (with Bedrock diagnosis)
IAM permissions needed: `bedrock:InvokeModel`, `sns:Publish`, plus the EKS access entry above for cluster actions.

```python
import boto3, json

bedrock = boto3.client("bedrock-runtime")
sns = boto3.client("sns")

def handler(event, context):
    # 1) Pull recent logs/metrics context for the alarm (CloudWatch Logs Insights query, abbreviated here)
    context_snippet = get_recent_logs_and_metrics()

    # 2) Ask Bedrock to classify — diagnosis only, no permissions to act
    prompt = f"""Classify this Kubernetes failure as one of:
    infrastructure, application, downstream_dependency, unknown.
    Context: {context_snippet}
    Respond with a single word and a confidence 0-1."""
    resp = bedrock.invoke_model(
        modelId="anthropic.claude-3-5-sonnet-20241022-v2:0",
        body=json.dumps({
            "anthropic_version": "bedrock-2023-05-31",
            "max_tokens": 20,
            "messages": [{"role": "user", "content": prompt}]
        })
    )
    classification, confidence = parse_classification(resp)

    # 3) Map classification to one fixed, pre-approved action
    if confidence < 0.7 or classification == "unknown":
        notify_only("Low-confidence classification, alerting instead of acting.")
    elif classification == "downstream_dependency":
        patch_configmap_feature_flag()
    elif classification == "application":
        rollout_restart_deployment()
    else:
        notify_only(f"Classified as {classification}, no automated action mapped.")
```

### 4.5 Wire CloudWatch Alarm → EventBridge → Lambda
- EventBridge Console → Rules → Create rule
- Event source: CloudWatch Alarm State Change, specific alarm `selfheal-restart-alarm`, state = ALARM
- Target: the remediation Lambda

### 4.6 Guardrails (don't skip this)
- **Bedrock gets read-only inputs and returns a label, nothing else.** It has no IAM permissions on EKS, Lambda, or anything that can act. The remediation Lambda is the only thing that executes, and only within the fixed action set (rollout restart, patch ConfigMap, notify)
- **Cooldown:** store the last remediation timestamp (DynamoDB or Parameter Store) and skip acting if within a cooldown window
- **Circuit breaker:** after N remediation attempts in a rolling window, stop acting automatically and notify only
- **Confidence threshold:** anything below it defaults to alert-only, never a guess

## Phase 5: Prove it with AWS FIS (native EKS actions)

### 5.1 Pod-level fault
```bash
aws fis create-experiment-template --cli-input-json '{
  "description": "Kill a selfheal-web pod",
  "targets": {
    "Pods": {
      "resourceType": "aws:eks:pod",
      "resourceTags": { "app": "selfheal-web" },
      "selectionMode": "COUNT(1)"
    }
  },
  "actions": {
    "podDelete": { "actionId": "aws:eks:pod-delete", "targets": { "Pods": "Pods" } }
  },
  "roleArn": "<fis-role-arn>",
  "stopConditions": [{ "source": "none" }]
}'
```

### 5.2 Node-level fault (managed node group only)
```bash
aws fis create-experiment-template --cli-input-json '{
  "description": "Terminate a node group instance",
  "targets": {
    "Nodegroup": {
      "resourceType": "aws:eks:nodegroup",
      "resourceArns": ["<nodegroup-arn>"],
      "selectionMode": "ALL"
    }
  },
  "actions": {
    "terminateNode": { "actionId": "aws:eks:terminate-nodegroup-instances", "parameters": { "instanceTerminationPercentage": "40" }, "targets": { "Nodegroup": "Nodegroup" } }
  },
  "roleArn": "<fis-role-arn>",
  "stopConditions": [{ "source": "none" }]
}'
```

### 5.3 Run both and record timestamps
Start each experiment from the FIS console (or `aws fis start-experiment`) and cross-reference recovery time against `kubectl get pods -o wide -w` and the CloudWatch dashboard in Phase 6.

## Notes for the write-up

- Every layer in this version is AWS-native: ALB ingress, EKS-managed node healing, CloudWatch/EventBridge/Lambda remediation, Bedrock diagnosis, FIS validation — nothing here depends on cluster-hosted tooling
- The scope boundary in 4.6 is the answer to "won't the AI just hallucinate and break something": Bedrock classifies, Lambda is the only thing with permission to act, and only within a fixed, pre-approved action set
- Capture the same evidence as before: screenshots/logs per phase, and real detection/recovery timestamps from the FIS runs
