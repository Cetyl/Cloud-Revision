# TaskBoard — Complete EKS Build Runbook

A ground-up, console-first runbook for building a 3-tier application on
Amazon EKS, covering all 13 phases in enough detail to recreate the
entire project from nothing. Built manually through the AWS Console
wherever possible (kubectl/Helm accepted for anything Kubernetes-native
that has no console equivalent), specifically to understand every layer
rather than abstract it away with `eksctl` or Terraform.

**Region:** `ap-south-1` (Mumbai) · **Account ID:** `397483229147` ·
**Cluster name:** `taskboard-cluster`

## 1. Architecture

```mermaid
flowchart TB
    DNS["Your DNS provider<br/>CNAME record"] --> CF["CloudFront distribution<br/>ACM cert · Phase 9"]
    CF --> ALB["ALB Ingress<br/>path routing · Phase 8"]

    subgraph VPC["VPC — 10.0.0.0/16"]
        direction TB
        NAT["NAT Gateway<br/>public subnet"]

        subgraph EKS["EKS cluster — taskboard-cluster (private subnets)"]
            direction TB
            ALB --> FSVC["Service: frontend-svc<br/>ClusterIP :80"]
            ALB --> BSVC["Service: backend-svc<br/>ClusterIP :4000"]
            FSVC --> FPods["Frontend pods x2<br/>nginx"]
            BSVC --> BPods["Backend pods x2<br/>node/express"]
            BPods --> EBS["EBS PVC<br/>Phase 7"]
            BPods --> EFS["EFS PVC (shared)<br/>Phase 7"]
            BPods --> SM["Secrets Manager CSI<br/>Phase 12"]
            HPA["HPA + Cluster Autoscaler<br/>Phase 10"] -.watches.-> BPods
            OBS["CloudWatch + Prometheus/Grafana<br/>Phase 11"] -.watches.-> EKS
            NP["NetworkPolicy<br/>Phase 12"] -.restricts.-> BPods
        end

        RDS[("RDS PostgreSQL<br/>taskboard-db")]
        BPods --> RDS
    end
```

Everything inside the EKS cluster stays on private subnets with no
direct internet exposure. The only entry point is the ALB (Phase 8),
itself only reachable through CloudFront once Phase 9 is done. The NAT
Gateway exists solely so private-subnet resources (nodes, pods) can
reach *out* to the internet (pulling images, calling AWS APIs) — nothing
reaches *in* through it.

## Phase 0 — IAM least-privilege setup

**Goal:** never touch this project as root or with `AdministratorAccess`.

1. **IAM Console → Users → Create user**
   - Name: `taskboard-eks-admin`
   - Check "Provide user access to the AWS Management Console"
   - Choose "I want to create an IAM user"
   - Set a custom password, require reset at next sign-in
   - Do **not** attach a policy yet — click through to create the user first

2. **Create the scoped policy — IAM → Policies → Create policy → JSON tab.**
   Paste a policy scoped to exactly this project's needs: `eks:*`; EC2
   networking/compute actions (VPC, subnet, IGW, NAT, route table,
   security group, EC2 instance/launch template, tagging including
   `DeleteTags`); `elasticloadbalancing:*`; ECR scoped to
   `arn:aws:ecr:*:*:repository/taskboard-*` plus the account-wide
   `ecr:GetAuthorizationToken`; `rds:*`; CloudWatch Logs/metrics/dashboards;
   Secrets Manager scoped to `arn:aws:secretsmanager:*:*:secret:taskboard-*`;
   ACM + CloudFront; IAM role create/attach/tag actions scoped to
   `arn:aws:iam::*:role/taskboard-*`, plus a separate `iam:PassRole`
   statement scoped to the same resource pattern with a
   `iam:PassedToService` condition restricted to
   `eks.amazonaws.com`, `ec2.amazonaws.com`, `eks-fargate-pods.amazonaws.com`.
   Name it `TaskBoard-EKS-Project-Policy`.
   **Naming rule for the rest of the project: every IAM role created
   must start with `taskboard-`**, since the policy is scoped to that
   prefix.

3. **Attach the policy** — IAM → Users → `taskboard-eks-admin` → Add
   permissions → Attach policies directly → `TaskBoard-EKS-Project-Policy`.

4. **Enable MFA** — user's Security credentials tab → Assign MFA device →
   Authenticator app → scan QR, confirm two consecutive codes.

5. **Generate access keys for CLI/kubectl/Docker** — same tab → Access
   keys → Create access key → use case "Command Line Interface (CLI)" →
   copy both values immediately (secret key is shown once).

6. **Configure locally:**
   ```bash
   aws configure
   # Access Key ID, Secret Access Key, region: ap-south-1, output: json
   aws sts get-caller-identity   # confirm ARN is taskboard-eks-admin, not root
   ```

7. **Sign in to the console as this user going forward** — never root,
   never a broader admin account, for the rest of this runbook.

## Phase 1 — Networking foundation

**Goal:** a custom VPC with correctly tagged public/private subnets.

1. **VPC Console → Your VPCs → Create VPC**
   - "VPC only" (not the wizard)
   - Name: `taskboard-vpc`, IPv4 CIDR: `10.0.0.0/16`, no IPv6, default tenancy
   - After creation: select it → Actions → Edit VPC settings → confirm
     both **Enable DNS resolution** and **Enable DNS hostnames** are checked
     (required for EKS)

2. **Create 4 subnets** — VPC Console → Subnets → Create subnet, all
   under `taskboard-vpc`:

   | Name | AZ | CIDR |
   |---|---|---|
   | `taskboard-public-1a` | ap-south-1a | `10.0.0.0/24` |
   | `taskboard-public-1b` | ap-south-1b | `10.0.1.0/24` |
   | `taskboard-private-1a` | ap-south-1a | `10.0.10.0/24` |
   | `taskboard-private-1b` | ap-south-1b | `10.0.11.0/24` |

3. **Enable auto-assign public IPv4** on both public subnets only
   (subnet → Actions → Edit subnet settings). Leave unchecked on both
   private subnets.

4. **Internet Gateway** — Create `taskboard-igw`, then Actions → Attach
   to VPC → `taskboard-vpc`.

5. **NAT Gateway** — Create `taskboard-nat` in `taskboard-public-1a`,
   connectivity type Public, allocate a new Elastic IP in the same step.

6. **Route tables:**
   - `taskboard-public-rt` → route `0.0.0.0/0` → `taskboard-igw` →
     associate both public subnets
   - `taskboard-private-rt` → route `0.0.0.0/0` → `taskboard-nat` →
     associate both private subnets

7. **Tag subnets for EKS — critical, do not skip:**

   Public subnets (`taskboard-public-1a`, `taskboard-public-1b`):
   ```
   kubernetes.io/cluster/taskboard-cluster = shared
   kubernetes.io/role/elb = 1
   ```
   Private subnets (`taskboard-private-1a`, `taskboard-private-1b`):
   ```
   kubernetes.io/cluster/taskboard-cluster = shared
   kubernetes.io/role/internal-elb = 1
   ```

8. **Sanity check** — VPC Console → your VPCs → `taskboard-vpc` →
   Resource map tab. Confirm 4 subnets correctly split, IGW attached,
   NAT sitting in a public subnet, both route tables associated correctly.

## Phase 2 — Cluster creation

1. **Cluster IAM role** — IAM → Roles → Create role → AWS service → use
   case **EKS - Cluster** → attaches `AmazonEKSClusterPolicy`
   automatically → name `taskboard-cluster-role`.

2. **Create the cluster** — EKS Console → Clusters → Create cluster:
   - Name: `taskboard-cluster`, latest stable Kubernetes version
   - Cluster service role: `taskboard-cluster-role`
   - Networking → VPC: `taskboard-vpc`; **Subnets: private subnets only**
     (`taskboard-private-1a`, `taskboard-private-1b`) — deliberately
     excludes public subnets; the ALB Controller (Phase 8) discovers
     subnets by VPC-wide tag, not by cluster association, so this
     doesn't block anything later
   - Cluster endpoint access: **Public and private** (public so you can
     kubectl from your laptop; private so node↔control-plane traffic
     stays inside the VPC)
   - Control plane egress: **AWS managed** (Customer routed is only
     needed for Hybrid Nodes / on-prem, not applicable here, and it's a
     one-way migration so leave the default)
   - Metrics: leave **Prometheus** and **CloudWatch** both **off** here —
     both get installed manually and deliberately in Phase 11 instead
   - Container network observability: **off**
   - Control plane logging: enable **all 5** — API server, Audit,
     Authenticator, Controller manager, Scheduler
   - Add-ons screen: select only **CoreDNS**, **kube-proxy**,
     **Amazon VPC CNI** — leave EBS CSI, EFS CSI, CloudWatch Observability,
     Load Balancer Controller, S3 CSI, node monitoring agent, GuardDuty
     agent, and Pod Identity Agent all unchecked (each gets added
     deliberately, later, with a manually-built IRSA role)
   - Create, wait ~10-15 minutes for **Active**

3. **Connect kubectl:**
   ```bash
   aws eks update-kubeconfig --region ap-south-1 --name taskboard-cluster
   kubectl get svc   # should show the default kubernetes ClusterIP service
   ```

4. **Create the OIDC provider manually** (this is the step that most
   often gets skipped since it's not automatic):
   ```bash
   aws eks describe-cluster --name taskboard-cluster --region ap-south-1 \
     --query "cluster.identity.oidc.issuer" --output text
   ```
   Copy the URL, then **IAM → Identity providers → Add provider →
   OpenID Connect** → paste the URL → **Get thumbprint** → Audience:
   `sts.amazonaws.com` → Add provider.

   Verify:
   ```bash
   aws iam list-open-id-connect-providers
   aws iam get-open-id-connect-provider --open-id-connect-provider-arn \
     arn:aws:iam::397483229147:oidc-provider/oidc.eks.ap-south-1.amazonaws.com/id/<oidc-id>
   # confirm "ClientIDList": ["sts.amazonaws.com"]
   ```

**Why the OIDC provider matters:** EKS auto-creates an OIDC *issuer*
(capable of signing tokens) the moment the cluster exists. But IAM won't
trust tokens from it until you separately register it as an IAM OIDC
*provider* — that's this step. Without it, IRSA (Phase 5 onward) cannot
work at all.

   attach `AmazonEKSWorkerNodePolicy`, `AmazonEKS_CNI_Policy`,
   `AmazonEC2ContainerRegistryReadOnly` → name `taskboard-node-role`.

2. **Managed node group** — EKS Console → taskboard-cluster → Compute →
   Add node group:
   - Name: `taskboard-nodegroup`, IAM role: `taskboard-node-role`
   - AMI: Amazon Linux 2023 (x86_64), instance type `t3.medium`, disk 20 GiB

## Phase 4 — Container images and ECR

**The app itself:** a 3-tier kanban board — React (Vite) frontend, Node/
Express backend with a `pg` (node-postgres) client, Postgres as the data
tier. Backend exposes `/api/tasks` (GET/POST/PATCH/DELETE) and `/health`
(deliberately DB-independent, used later as the liveness/readiness probe
target). Full source: `backend/src/{index,db,routes/tasks}.js`,
`frontend/src/{App,api,index.css,components/*}.jsx`.

**Dockerfiles:**
- **Backend**: single-stage `node:20-alpine`, `COPY package.json` +
  `npm install --omit=dev` *before* `COPY src` (maximizes layer cache
  reuse — code changes don't force a dependency reinstall), runs as a
  non-root user, exec-form `CMD ["node", "src/index.js"]` (ensures
  SIGTERM reaches the process directly for clean shutdowns).
- **Frontend**: multi-stage — Stage 1 (`node:20-alpine AS build`) runs
  `npm install` + `npm run build`, producing a `dist/` folder. Stage 2
  (`nginx:1.27-alpine`) does `COPY --from=build /app/dist ...` — pulling
  forward *only* the built static files, discarding `node_modules`,
  source, and all build tooling entirely. Final image ships nginx +
  static files only, ~40MB, no Node.js runtime at all.

1. **Create ECR repos** — ECR Console → Repositories → Create repository
   (Private) → `taskboard-frontend`, then again for `taskboard-backend`.
   Scan on push: optional but free, worth enabling.

2. **Authenticate Docker to ECR:**
   ```bash
   export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
   export AWS_REGION=ap-south-1
   export ECR_REGISTRY=$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com
   aws ecr get-login-password --region $AWS_REGION | \
     docker login --username AWS --password-stdin $ECR_REGISTRY
   ```

3. **Build and push — `--platform linux/amd64` is mandatory on Apple
   Silicon** (M1/M2 Macs build arm64 by default; the node group's EC2
   instances are x86_64 — without this flag the image pulls fine but
   crash-loops with an exec format error):
   ```bash
   docker build --platform linux/amd64 -t $ECR_REGISTRY/taskboard-backend:v1 ./backend
   docker push $ECR_REGISTRY/taskboard-backend:v1

   docker build --platform linux/amd64 -t $ECR_REGISTRY/taskboard-frontend:v1 ./frontend
   docker push $ECR_REGISTRY/taskboard-frontend:v1
   ```

4. **Verify:**
   ```bash
   aws ecr describe-images --repository-name taskboard-backend --region ap-south-1 \
     --query "imageDetails[].imageTags"
   aws ecr describe-images --repository-name taskboard-frontend --region ap-south-1 \
     --query "imageDetails[].imageTags"
   ```

**Test locally with docker-compose before ever touching ECR** — a
`docker-compose.yml` spinning up `postgres:16-alpine` + backend + frontend
together lets you verify the whole stack works in seconds, for free, before
any AWS resource exists. `docker compose up --build`, open
`http://localhost:8080`.

## Phase 5 — Add-ons and IRSA

1. **Simple add-ons** — EKS Console → taskboard-cluster → Add-ons →
   Get more add-ons → select **CoreDNS**, **kube-proxy**,
   **Amazon VPC CNI** together, default versions, Create. None need IAM.
   ```bash
   kubectl get pods -n kube-system   # coredns, aws-node, kube-proxy all Running
   ```

2. **Get OIDC details** (needed for every IRSA role from here on):
   - Account ID: shown in the console's top-right account menu
   - OIDC issuer ID: EKS Console → taskboard-cluster → Overview tab →
     "OpenID Connect provider URL" → the ID after `/id/`

3. **Create the IAM role for EBS CSI — Web Identity method:**
   - IAM → Roles → Create role → Trusted entity type: **Web identity**
   - Identity provider: select your cluster's OIDC provider from the dropdown
   - Audience: `sts.amazonaws.com`
   - Permissions: `AmazonEBSCSIDriverPolicy`
   - Name: `taskboard-ebs-csi-role`

4. **Tighten the trust policy to the exact service account** — this is
   the actual security-relevant IRSA mechanism:
   - Role → Trust relationships → Edit trust policy → ensure the
     `Condition` block reads:
   ```json
   "Condition": {
     "StringEquals": {
       "oidc.eks.ap-south-1.amazonaws.com/id/<oidc-id>:aud": "sts.amazonaws.com",
       "oidc.eks.ap-south-1.amazonaws.com/id/<oidc-id>:sub": "system:serviceaccount:kube-system:ebs-csi-controller-sa"
     }
   }
   ```
   The `:sub` condition is what restricts this role to *only* be
   assumable by that exact service account — without it, any pod with
   any service account and a validly-signed OIDC token could assume it.

5. **Install the add-on, selecting the role during creation (not after)** —
   Add-ons → Get more add-ons → Amazon EBS CSI Driver → Next → **expand
   "Optional configuration settings" explicitly** → IAM role:
   `taskboard-ebs-csi-role` → Next → Create.

6. **Repeat steps 3-5 for EFS CSI**, using `AmazonEFSCSIDriverPolicy` and
   `:sub` = `system:serviceaccount:kube-system:efs-csi-controller-sa`,
   role name `taskboard-efs-csi-role`.

7. **Verify both:**
   ```bash
   kubectl get pods -n kube-system | grep -E 'ebs-csi|efs-csi'   # all Running, full readiness
   kubectl describe sa ebs-csi-controller-sa -n kube-system
   kubectl describe sa efs-csi-controller-sa -n kube-system
   # both should show: Annotations: eks.amazonaws.com/role-arn: arn:aws:iam::...:role/taskboard-...-csi-role
   ```

   **Known console reliability issue:** the add-on wizard sometimes
   fails to persist the IAM role selection even when it was visibly
   selected — service accounts show `Annotations: <none>` and controller
   pods crash-loop with partial readiness (e.g. `1/6`). If this happens:
   remove and recreate the add-on, watching the role dropdown carefully;
   if it happens again, set the annotation directly and force a restart:
   ```bash
   kubectl annotate serviceaccount ebs-csi-controller-sa -n kube-system \
     eks.amazonaws.com/role-arn=arn:aws:iam::397483229147:role/taskboard-ebs-csi-role --overwrite
   kubectl delete pod -n kube-system -l app.kubernetes.io/name=aws-ebs-csi-driver
   ```
   (Same pattern for EFS, substituting the EFS role/label.) See the
   [Bugs](#bugs-encountered-and-fixes) section for the full writeup.

## Phase 6 — Deploy the app

1. **DB subnet group** — RDS Console → Subnet groups → Create →
   `taskboard-db-subnet-group`, VPC `taskboard-vpc`, both private subnets.

2. **RDS security group** — VPC Console → Security groups → Create →
   `taskboard-rds-sg`, VPC `taskboard-vpc`. Inbound: PostgreSQL (5432),
   **source = your node group's security group** (EKS Console →
   taskboard-cluster → Compute → node group → Details — usually named
   `eks-cluster-sg-taskboard-cluster-...`). This is the actual mechanism
   that lets pods reach RDS — pods inherit node security groups via the
   VPC CNI; it isn't about subnet proximity.

3. **Create the RDS instance** — RDS Console → Create database:
   - Engine PostgreSQL, latest stable, template Free tier/Dev-Test
   - Identifier `taskboard-db`, master username `taskboard`, set a password
   - Instance class `db.t3.micro`, 20 GiB gp3, Multi-AZ **No**
   - VPC `taskboard-vpc`, DB subnet group `taskboard-db-subnet-group`
   - Public access **No**, security group → existing → `taskboard-rds-sg`
   - **Initial database name: `taskboard`** — do not leave this blank;
     skipping it means only the default `postgres` database exists and
     your app will fail with `database "taskboard" does not exist`
   - Create, wait 5-10 min, copy the endpoint once Available

4. **Namespace and Secret:**
   ```bash
   kubectl create namespace taskboard
   kubectl create secret generic taskboard-db-secret \
     --namespace taskboard --from-literal=PGPASSWORD='<your-rds-password>'
   ```

5. **Backend Deployment + Service** (`backend-deployment.yaml`):
   ```yaml
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: taskboard-backend
     namespace: taskboard
   spec:
     replicas: 2
     selector:
       matchLabels:
         app: taskboard-backend
     template:
       metadata:
         labels:
           app: taskboard-backend
       spec:
         containers:
           - name: backend
             image: 397483229147.dkr.ecr.ap-south-1.amazonaws.com/taskboard-backend:v2
             ports:
               - containerPort: 4000
             env:
               - name: PGHOST
                 value: "<your-rds-endpoint>"
               - name: PGPORT
                 value: "5432"
               - name: PGUSER
                 value: "taskboard"
               - name: PGDATABASE
                 value: "taskboard"
               - name: PORT
                 value: "4000"
               - name: PGSSL
                 value: "true"
               - name: PGPASSWORD
                 valueFrom:
                   secretKeyRef:
                     name: taskboard-db-secret
                     key: PGPASSWORD
             readinessProbe:
               httpGet: { path: /health, port: 4000 }
               initialDelaySeconds: 5
               periodSeconds: 10
             livenessProbe:
               httpGet: { path: /health, port: 4000 }
               initialDelaySeconds: 15
               periodSeconds: 20
             resources:
               requests: { cpu: "100m", memory: "128Mi" }
               limits: { cpu: "300m", memory: "256Mi" }
   ---
   apiVersion: v1
   kind: Service
   metadata:
     name: taskboard-backend-svc
     namespace: taskboard
   spec:
     selector:
       app: taskboard-backend
     ports:
       - port: 4000
         targetPort: 4000
   ```
   (`PGSSL: "true"` and image tag `:v2` reflect a fix from the first
   build — see Bugs section. `readinessProbe` controls whether traffic
   routes to the pod; `livenessProbe` controls whether Kubernetes
   restarts it. Both hit `/health`, which never touches the DB — a slow
   RDS shouldn't cause Kubernetes to think the *pod* is broken.)

6. **Frontend Deployment + Service** (`frontend-deployment.yaml`) — same
   shape, no `env` block, no probes, `containerPort: 80`,
   `image: .../taskboard-frontend:v3` (tag reflects the nginx DNS fix,
   see Bugs), Service `port: 80`/`targetPort: 80`. Both Services default
   to `ClusterIP` (no `type:` field) — deliberately not internet-facing
   yet; Phase 8's ALB Ingress is the intended external entry point.

7. **Apply and verify:**
   ```bash
   kubectl apply -f backend-deployment.yaml
   kubectl apply -f frontend-deployment.yaml
   kubectl get pods -n taskboard
   kubectl logs deployment/taskboard-backend -n taskboard   # look for "Database schema ready"
   kubectl port-forward svc/taskboard-frontend-svc -n taskboard 8080:80
   # open http://localhost:8080, add a task, confirm it persists
   ```

## Phase 7 — Storage

**Assumed starting point:** you're in the same folder you've been running
`kubectl apply -f` from for Phase 6 — the one containing
`backend-deployment.yaml` and `frontend-deployment.yaml`. If you're not
sure, run `pwd` to see where you are, and `ls` to confirm those two files
are there. Everything below happens in this same folder.

```bash
cd ~/Documents/Projects/taskboard/deployment-files   # adjust to your actual path
ls   # confirm backend-deployment.yaml and frontend-deployment.yaml are here
```

### Part A: EBS PVC (per-pod storage)

**Step 1 — create a brand new file named `ebs-storageclass.yaml`.**
This is a *new* file — it does not exist yet. Run this entire block
exactly as-is in your terminal; it creates the file and writes the
content into it in one action, no text editor needed:

```bash
cat > ebs-storageclass.yaml << 'EOF'
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: taskboard-gp3
provisioner: ebs.csi.aws.com
volumeBindingMode: WaitForFirstConsumer
parameters:
  type: gp3
EOF
```

Confirm it was created correctly:
```bash
ls                              # ebs-storageclass.yaml should now appear
cat ebs-storageclass.yaml       # should print exactly what you just wrote
```

`WaitForFirstConsumer` delays actually creating the EBS volume until a
pod using it gets scheduled, so the volume lands in the same
Availability Zone as that pod. Without this, the volume could get
created in the wrong AZ and the pod would sit stuck in `Pending` forever.

**Step 2 — create a second new file named `ebs-pvc.yaml`.** Same
technique, different file, different content:

```bash
cat > ebs-pvc.yaml << 'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: taskboard-backend-data
  namespace: taskboard
spec:
  accessModes: ["ReadWriteOnce"]
  storageClassName: taskboard-gp3
  resources:
    requests:
      storage: 5Gi
EOF
```

```bash
ls                # ebs-pvc.yaml should now also appear
cat ebs-pvc.yaml
```

**Step 3 — apply both new files to the cluster:**
```bash
kubectl apply -f ebs-storageclass.yaml
kubectl apply -f ebs-pvc.yaml
```

**Step 4 — verify:**
```bash
kubectl get storageclass
# taskboard-gp3 should be listed

kubectl get pvc -n taskboard
# taskboard-backend-data should show STATUS: Pending
# (Pending is CORRECT and expected right now - it only becomes Bound

```

**Step 5 — this is different: you are EDITING a file you already have,
not creating a new one.** Rather than hand-editing individual lines in
nano (very easy to get the indentation wrong — see Bug #6 below, hit
twice during this exact step), **overwrite the whole file** with the
corrected version in one shot. This adds the `volumeMounts` block, the
`volumes` block, **and** a `securityContext.fsGroup` setting — the
fsGroup is required because the backend container runs as a non-root
user (Dockerfile hardening from Phase 4), and a freshly-provisioned EBS
volume is root-owned by default; without `fsGroup`, the container gets
a `Permission denied` error the first time it tries to write to the
mounted volume (see Bug #7). `101` is this project's actual container
group ID — confirm yours matches with `kubectl exec -it <any-backend-pod> -n taskboard -- id`
before assuming it's the same:

```bash
cat > backend-deployment.yaml << 'EOF'
apiVersion: apps/v1
kind: Deployment
metadata:
  name: taskboard-backend
  namespace: taskboard
spec:
  replicas: 2
  selector:
    matchLabels:
      app: taskboard-backend
  template:
    metadata:
      labels:
        app: taskboard-backend
    spec:
      securityContext:
        fsGroup: 101
      containers:
        - name: backend
          image: 397483229147.dkr.ecr.ap-south-1.amazonaws.com/taskboard-backend:v2
          ports:
            - containerPort: 4000
          env:
            - name: PGHOST
              value: "<your-rds-endpoint>"
            - name: PGPORT
              value: "5432"
            - name: PGUSER
              value: "taskboard"
            - name: PGDATABASE
              value: "taskboard"
            - name: PORT
              value: "4000"
            - name: PGSSL
              value: "true"
            - name: PGPASSWORD
              valueFrom:
                secretKeyRef:
                  name: taskboard-db-secret
                  key: PGPASSWORD
          readinessProbe:
            httpGet:
              path: /health
              port: 4000
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 4000
            initialDelaySeconds: 15
            periodSeconds: 20
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "300m"
              memory: "256Mi"

            - name: backend-data
              mountPath: /app/data

        - name: backend-data

apiVersion: v1
kind: Service

  namespace: taskboard
spec:
  selector:
    app: taskboard-backend
  ports:
    - port: 4000

```

**Before applying**, sanity-check the indentation by eye:
```bash
cat backend-deployment.yaml
```
`volumeMounts:` must line up with `resources:` (siblings, both children
of the `backend` container) — **not** nested one level deeper as if it
were a child of `resources:`. `volumes:` must line up with `containers:`
(siblings, both children of `spec:` under `template:`) — **not** floating
between `template.metadata` and `template.spec`. Getting either of these
wrong is the single most common mistake at this step (see Bug #6).

**Step 6 — apply the updated file and verify:**
```bash
kubectl apply -f backend-deployment.yaml
kubectl get pods -n taskboard -w
# wait for backend pods to restart and reach Running (Ctrl+C to stop watching)

kubectl get pvc -n taskboard
# taskboard-backend-data should now show STATUS: Bound
```

**Step 7 — final proof it actually works:**
```bash
kubectl get pods -n taskboard -l app=taskboard-backend
# copy one pod name from the output

kubectl exec -it <paste-pod-name-here> -n taskboard -- sh -c "echo test123 > /app/data/check.txt && cat /app/data/check.txt"
# should print: test123
# if you instead get "Permission denied": your container's non-root
# user/group doesn't own the volume - run `kubectl exec -it <pod> -n
```

### Part B: EFS (shared across pods)

**Step 1 — create the filesystem itself (console, not a file on your laptop).**
Go to **EFS Console → File systems → Create file system**:
- Name: `taskboard-efs`
- VPC: `taskboard-vpc`
- Under "Network access," confirm mount targets are being created in
  both `taskboard-private-1a` and `taskboard-private-1b`
- Create

Wait for the filesystem's **Life cycle state** to show **Available**,
then copy its **File system ID** (looks like `fs-0123456789abcdef0`) —
you'll paste this into a file in Step 4.

**Step 2 — create the EFS security group (console).**
**VPC Console → Security groups → Create security group:**
- Name: `taskboard-efs-sg`, VPC: `taskboard-vpc`
- Inbound rule: Type **NFS** (auto-fills port 2049), Source: your node
  group's security group (same one used for the RDS security group back
  in Phase 6 — find it again at EKS Console → taskboard-cluster →
  Compute → node group → Details if you need to)
- Create

**Step 3 — attach this security group to the EFS mount targets.**
Back in **EFS Console → taskboard-efs → Network tab**, edit each mount
target and add `taskboard-efs-sg` to it (in addition to, or replacing,
whatever default security group is there).

**Step 4 — create a new file named `efs-storageclass.yaml`** on your
laptop, in the same `deployment-files` folder as before:

```bash
cat > efs-storageclass.yaml << 'EOF'
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: taskboard-efs-sc
provisioner: efs.csi.aws.com
parameters:
  provisioningMode: efs-ap
  fileSystemId: fs-XXXXXXXX
  directoryPerms: "700"
EOF
```

**Before applying this**, edit the file to replace `fs-XXXXXXXX` with
your actual File system ID from Step 1:
```bash
nano efs-storageclass.yaml
# change fs-XXXXXXXX to your real ID, e.g. fs-0123456789abcdef0
# save and close
```

**Step 5 — create a new file named `efs-pvc.yaml`:**
```bash
cat > efs-pvc.yaml << 'EOF'
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: taskboard-shared-storage
  namespace: taskboard
spec:
  accessModes: ["ReadWriteMany"]
  storageClassName: taskboard-efs-sc
  resources:
    requests:
      storage: 5Gi
EOF
```

Notice `ReadWriteMany` here, versus `ReadWriteOnce` for the EBS PVC
above — that's the entire point of using EFS: multiple pods, even on
different nodes, can mount this at the same time.

**Step 6 — apply both:**
```bash
kubectl apply -f efs-storageclass.yaml
kubectl apply -f efs-pvc.yaml
kubectl get pvc -n taskboard
# taskboard-shared-storage should appear, eventually reaching Bound
# (may take a minute or two longer than EBS did)
# If it stays Pending with a "no EC2 IMDS role found" / credential
# error in `kubectl describe pvc taskboard-shared-storage -n taskboard`,
# see Bug #8 - the fix is restarting the EFS CSI controller pods, even
# if their service account annotation already looks correct.
```

**Step 7 — edit `backend-deployment.yaml` again** (same file, third edit
overall now) to add a second volume mount. Under the `volumeMounts:`
block you added in Part A Step 5, add one more entry so it reads:
```yaml
          volumeMounts:
            - name: backend-data
              mountPath: /app/data
            - name: shared-data
              mountPath: /app/shared
```
And under the `volumes:` block from Part A, add a second entry:
```yaml
      volumes:
        - name: backend-data
          persistentVolumeClaim:
            claimName: taskboard-backend-data
        - name: shared-data
          persistentVolumeClaim:
            claimName: taskboard-shared-storage
```

**Step 8 — apply and do the real cross-pod proof:**
```bash
kubectl apply -f backend-deployment.yaml
kubectl get pods -n taskboard -w   # wait for Running, Ctrl+C when done

kubectl get pods -n taskboard -l app=taskboard-backend
# you'll see two pod names - copy both
```

Write a file from the **first** pod:
```bash
kubectl exec -it <pod-1-name> -n taskboard -- sh -c "echo hello-from-pod-1 > /app/shared/test.txt"
```

Read it back from the **second** pod:
```bash
kubectl exec -it <pod-2-name> -n taskboard -- cat /app/shared/test.txt

```

## Phase 8 — Ingress and load balancing

**Step 1 — get the IAM policy JSON you need (this is not something you
write yourself; you download AWS's official, maintained version).**
```bash
curl -o iam_policy.json https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.9.0/docs/install/iam_policy.json
cat iam_policy.json   # just to see it's real content, not an error page
```

**Step 2 — create the IAM policy from this file (console).**
IAM Console → Policies → Create policy → **JSON tab** → delete the
placeholder text in the box → paste the entire contents of
`iam_policy.json` (open it with `cat iam_policy.json` and copy everything
it prints) → Next → name it `taskboard-alb-controller-policy` → Create policy.

**Step 3 — get your OIDC ID again** (same value you've used since Phase 5,
but if you rebuilt the cluster since, get the current one):
```bash
aws eks describe-cluster --name taskboard-cluster --region ap-south-1 \
  --query "cluster.identity.oidc.issuer" --output text
# copy the part after "/id/"
```

**Step 4 — create the IRSA role (console), same Web Identity pattern as
Phase 5:**
- IAM → Roles → Create role → Trusted entity type: **Web identity**
- Identity provider: your cluster's OIDC provider
- Audience: `sts.amazonaws.com`
- Next → search for and attach `taskboard-alb-controller-policy` (the
  one you just created in Step 2)
- Name: `taskboard-alb-controller-role`
- Create role

**Step 5 — tighten the trust policy** (same as every IRSA role so far):
IAM → Roles → `taskboard-alb-controller-role` → Trust relationships →
Edit trust policy → confirm/set the `Condition` block:
```json
"Condition": {
  "StringEquals": {
    "oidc.eks.ap-south-1.amazonaws.com/id/<your-oidc-id>:aud": "sts.amazonaws.com",
    "oidc.eks.ap-south-1.amazonaws.com/id/<your-oidc-id>:sub": "system:serviceaccount:kube-system:aws-load-balancer-controller"
  }
}
```
Replace `<your-oidc-id>` with the value from Step 3, in both places.

**Step 6 — get your VPC ID** (needed for the Helm install command below):
```bash
aws ec2 describe-vpcs --filters "Name=tag:Name,Values=taskboard-vpc" \
  --query "Vpcs[0].VpcId" --output text
# copy this value, looks like vpc-0123456789abcdef0
```

**Step 7 — install the controller via Helm** (you already have Helm
installed from your initial Mac setup; confirm with `helm version` if unsure):
```bash
helm repo add eks https://aws.github.io/eks-charts
helm repo update
```

Then run this, replacing `<your-vpc-id>` with the value from Step 6:
```bash
helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
  -n kube-system \
  --set clusterName=taskboard-cluster \
  --set region=ap-south-1 \
  --set vpcId=<your-vpc-id> \
  --set serviceAccount.create=true \
  --set serviceAccount.name=aws-load-balancer-controller \
  --set serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::397483229147:role/taskboard-alb-controller-role
```

**Step 8 — verify the controller itself is running before creating any Ingress:**
```bash
kubectl get deployment -n kube-system aws-load-balancer-controller
# should show READY 2/2 (or 1/1 depending on replica count)

kubectl logs -n kube-system deployment/aws-load-balancer-controller
# should show normal startup logs, no repeated error messages
```

**Step 9 — create a brand new file named `taskboard-ingress.yaml`**, back
in your `deployment-files` folder:
```bash
cat > taskboard-ingress.yaml << 'EOF'
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: taskboard-ingress
  namespace: taskboard
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/scheme: internet-facing
    alb.ingress.kubernetes.io/target-type: ip
    alb.ingress.kubernetes.io/healthcheck-path: /health
spec:
  rules:
    - http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service:
                name: taskboard-backend-svc
                port:
                  number: 4000
          - path: /
            pathType: Prefix
            backend:
              service:
                name: taskboard-frontend-svc
                port:
                  number: 80
EOF
```

`healthcheck-path: /health` matters even though it's easy to skip: the
ALB Controller defaults to health-checking `/`, which the backend has
no route for at all (only `/health` and `/api/tasks` exist) — without
this annotation the backend target group shows Unhealthy [404] even
though the app works perfectly. See Bug #10.

**Before applying — double-check both public subnets' tags one more
time.** This is the step where a subnet tagging mistake from Phase 1
(`elb` vs `internal-elb` swapped) first becomes visible — everything
before this phase works fine even with a wrong tag, since nothing else
needs the ALB Controller's subnet auto-discovery. See Bug #9.
```bash
aws ec2 describe-subnets \
  --filters "Name=tag:Name,Values=taskboard-public-1a,taskboard-public-1b" \
  --query "Subnets[].Tags" --region ap-south-1 --output table
```
Both should show `kubernetes.io/role/elb = 1` (not `internal-elb`) and
`kubernetes.io/cluster/taskboard-cluster = shared`.

**Step 10 — apply it:**
```bash
kubectl apply -f taskboard-ingress.yaml
```

**If the Ingress's `ADDRESS` stays empty for more than a few minutes**,
check `kubectl describe ingress taskboard-ingress -n taskboard` for a
`FailedBuildModel` event — and if you fix a subnet tag *after* already
applying the Ingress, the controller won't automatically notice; force
an immediate re-check with:
```bash
kubectl annotate ingress taskboard-ingress -n taskboard force-refresh="$(date +%s)" --overwrite
```

**Step 11 — wait and verify** (ALB provisioning takes a few minutes):
```bash
kubectl get ingress -n taskboard -w
# watch the ADDRESS column - it starts empty, then fills in with
# something like taskboard-ingress-xxxxx.ap-south-1.elb.amazonaws.com
# Ctrl+C once ADDRESS is populated
```

**Step 12 — test it actually works, from your laptop, no port-forward needed this time:**
```bash
curl http://<paste-the-address-from-step-11>/
curl http://<paste-the-address-from-step-11>/api/health
```
The first should return your frontend's HTML; the second should return
`{"status":"ok"}`. If both work, your app is now reachable from the
internet through a real load balancer for the first time in this project.

## Phase 9 — CDN and custom domain

**Step 1 — switch your AWS Console region to N. Virginia (us-east-1)
temporarily.** Use the region dropdown in the top-right of the console.
This is not a mistake — CloudFront specifically requires its certificate
to exist in `us-east-1` regardless of where your actual cluster lives.

**Step 2 — request the certificate.** ACM Console (while still in
us-east-1) → Request certificate → Request a public certificate → Next:
- Fully qualified domain name: your subdomain, e.g. `taskboard.yourdomain.com`
- Validation method: **DNS validation**
- Request

**Step 3 — add the validation record at your own DNS provider.** Click
into the newly requested certificate, find the **CNAME name** and
**CNAME value** it shows you, and add exactly that as a new CNAME record
at whatever DNS provider you use (GoDaddy, Cloudflare, Namecheap, etc. —
the exact steps differ per provider, but every provider has an "Add
record" button in their DNS management page).

**Step 4 — wait for validation.** Refresh the ACM certificate page every
so often; status changes from "Pending validation" to **"Issued"** once
your DNS provider's change propagates (can be minutes to a few hours).

**Step 5 — switch your console region back to ap-south-1** (or leave a
second browser tab on us-east-1 for the next CloudFront step, since
CloudFront itself is a global service and its console is reachable from
any region).

**Step 6 — create the CloudFront distribution.** CloudFront Console →
Create distribution:
- Origin domain: paste your ALB's address (the same one from Phase 8,
  Step 11 — e.g. `taskboard-ingress-xxxxx.ap-south-1.elb.amazonaws.com`)
- Origin protocol policy: **HTTP only** (your ALB has no HTTPS listener
  yet — that's a Phase 12+ hardening exercise, not required now)
- Viewer protocol policy: **Redirect HTTP to HTTPS**
- Add a second cache behavior for path pattern `/api/*`: cache policy
  **CachingDisabled**, origin request policy **AllViewer** (forwards all
  headers/cookies through, since API responses shouldn't be cached like
  static assets)
- Default cache behavior (matches everything else, i.e. `/*`): cache
  policy **CachingOptimized**
- Alternate domain name (CNAME): your subdomain from Step 2
- Custom SSL certificate: select the one you created (it should now
  appear in the dropdown since it's Issued)
- Create distribution

**Step 7 — wait for the distribution to finish deploying** (Status
column flips from "Deploying" to a blank/green "Enabled" state — can
take 10-15 minutes).

**Step 8 — point your own domain at CloudFront.** Back at your DNS
provider, add a new record:
- Type: **CNAME** (or **ALIAS**/**ANAME** if your provider offers it and
  your domain is a root domain rather than a subdomain)
- Name: your subdomain (e.g. `taskboard`)
- Value: the CloudFront distribution's domain name, shown on the
  distribution's overview page, looks like `d1234abcdwxyz.cloudfront.net`

**Step 9 — verify:**
```bash
dig taskboard.yourdomain.com     # or: nslookup taskboard.yourdomain.com
curl -I https://taskboard.yourdomain.com
# look for an X-Cache header in the response - its presence confirms
# CloudFront is actually in the request path, not just DNS pointing at it
```

## Bugs encountered and fixes

### 1. RDS SSL enforcement rejecting the backend's connection
**Symptom:** `no pg_hba.conf entry for host "10.0.10.5"... no encryption`
**Cause:** RDS Postgres enforces SSL by default; the `pg` library
defaults to no SSL. Never surfaced locally since docker-compose Postgres
doesn't enforce SSL at all.
**Fix:** `ssl: process.env.PGSSL === 'true' ? { rejectUnauthorized: false } : false`
in `db.js`'s `Pool` config; set `PGSSL=true` only in the Kubernetes
Deployment env, leaving local docker-compose untouched. Rebuilt as `:v2`.
*(Phase 12+ hardening: validate against the real RDS CA bundle instead
of `rejectUnauthorized: false`.)*

### 2. Target database never created inside the RDS instance
**Symptom:** `database "taskboard" does not exist`
**Cause:** "Initial database name" left blank during RDS creation — the
master **user** existed, the **database** did not.
**Fix:** Temporary self-deleting pod to run SQL from inside the VPC
(RDS is private-only, unreachable from a laptop directly):
```bash
kubectl run psql-client --rm -it --restart=Never -n taskboard \
  --image=postgres:16-alpine -- \
  psql "postgresql://taskboard:<password>@<rds-endpoint>:5432/postgres?sslmode=require"
```
then `CREATE DATABASE taskboard OWNER taskboard;`. This pod could reach
RDS with zero extra security-group changes because pods inherit their
node's security groups via the VPC CNI, and the node group's SG was
already allow-listed on the RDS SG. **Prevention:** always fill in
"Initial database name" during RDS creation (see Phase 6, step 3).

### 3. nginx crash-looping / 502 due to a docker-compose-only hostname
**Symptom (boot crash):** `nginx: [emerg] host not found in upstream "backend"`
**Symptom (post-partial-fix, on use):** `502 Bad Gateway` when adding a task
**Cause:** `nginx.conf`'s `/api/` proxy block pointed at `backend:4000` —
a hostname only meaningful inside docker-compose's private network.
Static `proxy_pass` targets resolve once at nginx startup; failed
resolution crashes nginx entirely. In Kubernetes, `backend` doesn't
resolve to anything.
**Fix (two-part):** (1) switched to a variable-based `proxy_pass`
(`set $upstream_backend ...;`) so resolution happens per-request instead
of at boot, stopping the crash; (2) pointed the variable at the real
Kubernetes Service DNS name — `taskboard-backend-svc.taskboard.svc.cluster.local:4000`
— with an explicit `resolver` directive using CoreDNS's ClusterIP
(`kubectl get svc kube-dns -n kube-system`). Rebuilt as `:v3`. Note: this
block only matters for `kubectl port-forward` testing — Phase 8's ALB
intercepts `/api/*` before it reaches the frontend pod in production.

### 4. EKS add-on console UI not persisting the IRSA role selection
**Symptom:** service account shows `Annotations: <none>` after selecting
an IAM role during EBS/EFS CSI Driver add-on setup; controller pods
crash-loop with partial readiness (e.g. `1/6`) and credential errors.
Recurred on both drivers, at both the original setup and again after a
full cluster rebuild.
**Root cause:** apparent console reliability issue — the add-on wizard
silently fails to write the `eks.amazonaws.com/role-arn` annotation even
when the role was visibly selected pre-Create.
**Fix:** set the annotation directly and force pod recreation:
```bash
kubectl annotate serviceaccount ebs-csi-controller-sa -n kube-system \
  eks.amazonaws.com/role-arn=arn:aws:iam::397483229147:role/taskboard-ebs-csi-role --overwrite
kubectl delete pod -n kube-system -l app.kubernetes.io/name=aws-ebs-csi-driver
```
(mirror for EFS). Ruled out along the way: trust policy correctness
(confirmed correct via IAM JSON view both times) and node schedulability
(`InsufficientNumberOfReplicas` briefly appeared mid-rebuild simply
because the node group hadn't finished initializing — resolved once
`kubectl get nodes` showed `Ready`, no action needed).

### 5. Cost-driven full EKS teardown and rebuild
Not a bug — a deliberate exercise. EKS control plane has no stop/pause
option (delete-or-nothing), so it was fully deleted (Fargate profile →
node group → cluster → OIDC provider, in that order) to stop billing
overnight. **RDS was stopped, not deleted** — RDS supports up to a 7-day
stop, preserving data, master user, and endpoint hostname, at far less
effort than deletion. Rebuilding EKS required: a new cluster, a **new**
OIDC provider (new clusters mint a new OIDC ID), **updating both CSI
IRSA roles' trust policies** with the new ID, recreating the node group
and Fargate profile, and re-hitting (and re-fixing, via the same kubectl
annotate workaround) bug #4 on both CSI drivers. Because RDS was only
stopped, no database/security-group/`PGHOST` changes were needed there.

### 6. YAML indentation errors when adding volumeMounts/volumes to an existing Deployment
**Symptom:** `kubectl apply` succeeded (no error) but the file was
structurally wrong in two ways across two attempts: (1) a `volumes:`
block landed between `template.metadata` and `template.spec` instead of
as a sibling of `containers:` under `template.spec`; (2) `volumeMounts:`
landed nested one level too deep, as if it were a child of `resources:`
instead of a sibling of it under the `backend` container.
**Cause:** hand-editing an existing multi-level YAML file in nano,
adding new blocks by eye without a strict reference for exact
indentation depth. YAML has no closing tags to catch this the way
XML/HTML would — a misplaced block simply gets attached to the wrong
parent silently.
**Fix:** stopped hand-editing individual lines and instead overwrote the
entire file in one shot via `cat > file.yaml << 'EOF' ... EOF`, with
every line's indentation fixed in the heredoc itself — removing any
chance of a manual spacing slip. **Prevention going forward:** for any
Deployment edit requiring new nested blocks (volumes, env, probes,
etc.), prefer a full-file heredoc overwrite over manual mid-file edits,
and always `cat` the file back afterward to visually confirm structure
before running `kubectl apply`.

### 7. EBS volume "Permission denied" for a non-root container
**Symptom:** `sh: can't create /app/data/check.txt: Permission denied`
when writing to a freshly-mounted EBS-backed PVC, even though the PVC
showed `STATUS: Bound` and the pod was `Running` with no errors.
**Cause:** the backend container deliberately runs as a non-root user
(`appuser`, gid `101` — hardening from the Phase 4 Dockerfile), but a
newly provisioned EBS volume is root-owned by default. The container
had no write permission on its own mounted volume.
**Fix:** added `securityContext: { fsGroup: 101 }` at the pod level
(sibling of `containers:`, under `template.spec`) — this tells
Kubernetes to recursively set that group as the owner of any mounted
volume for this pod, before the container starts. Found the correct gid
via `kubectl exec -it <pod> -n taskboard -- id`. This doesn't weaken the
non-root container hardening at all; it just grants the *group* write
access to the volume, which is the standard, correct fix for this exact
mismatch.

### 8. EFS CSI controller crashing on provisioning despite a correct IRSA annotation
**Symptom:** `taskboard-shared-storage` PVC stuck `Pending`;
`kubectl describe pvc` showed `ProvisioningFailed... get identity: get
credentials: failed to refresh cached credentials, no EC2 IMDS role
found`. Unlike bug #4, `kubectl describe sa efs-csi-controller-sa -n
kube-system` showed the **correct** `eks.amazonaws.com/role-arn`
annotation already in place — so this looked identical to the earlier
IRSA bug on the surface, but the root cause was different.
**Cause:** the Pod Identity Webhook only injects AWS credentials (env
vars + a projected OIDC token volume) into a pod **at the moment it's
created**. The EFS CSI controller pods had been running for hours
(since before the annotation was last corrected), so they were still
running with zero AWS credentials even though the service account
object itself now looked correctly configured. A correct annotation on
the ServiceAccount does nothing for pods that already existed before it
was set.
**Fix:**
```bash
kubectl delete pod -n kube-system -l app.kubernetes.io/name=aws-efs-csi-driver
```
forcing fresh pods to be created, which then correctly picked up the
already-correct annotation. **General lesson:** whenever an IRSA-backed
pod is crash-looping or erroring on credentials, check both things
separately — (1) is the ServiceAccount's annotation actually correct,
and (2) were the current running pods created *after* that annotation
was set. Fixing the annotation alone does not help pods already running
before the fix.

### 9. ALB never provisioned: one public subnet mistagged, and tag fixes don't auto-trigger reconciliation
**Symptom:** `kubectl get ingress -n taskboard` showed an empty
`ADDRESS` column for over an hour. `kubectl describe ingress` showed a
repeating event: `Failed build model due to couldn't auto-discover
subnets: subnets count less than minimal required count: 1 < 2`.
**Cause:** `taskboard-public-1a` was tagged
`kubernetes.io/role/internal-elb = 1` (the tag meant for **private**
subnets) instead of `kubernetes.io/role/elb = 1` (the tag an
internet-facing ALB actually looks for on **public** subnets) — a
copy-paste mix-up from Phase 1, undetected until this phase because
nothing before Phase 8 actually needed the ALB Controller to discover
subnets by tag. `taskboard-public-1b` had the correct tag the whole
time, which is why the count showed "1" found instead of "0."
**Fix (two parts):** (1) corrected the tag on `taskboard-public-1a` via
console (VPC → Subnets → Tags tab); (2) this alone did not resolve the
Ingress — the ALB Controller does not automatically re-scan subnets
just because a tag changed elsewhere in the VPC; it only reconciles on
its own schedule or when the Ingress object itself changes. Forced an
immediate reconcile with:
```bash
kubectl annotate ingress taskboard-ingress -n taskboard force-refresh="$(date +%s)" --overwrite
```
which triggered `Successfully reconciled` within about a minute, and
the ALB's real DNS address appeared. **Prevention:** double-check both
public subnets' tags immediately after Phase 1, specifically confirming
`elb` vs `internal-elb` are not swapped, rather than waiting until
Phase 8 to discover a mismatch.

### 10. ALB target groups showing Unhealthy [404] despite the app working correctly
**Symptom:** the app was reachable and fully functional via
`kubectl port-forward` and even via direct `curl` through the ALB's own
DNS name, yet the EC2 Console's Target Groups / load balancer resource
map showed the **backend** target group's targets as
`Unhealthy: Health checks failed with these codes: [404]`, while the
frontend target group showed `Healthy`.
**Cause:** the ALB Controller defaults to health-checking path `/`
unless told otherwise. The frontend (nginx) legitimately returns 200 at
`/`. The backend (Express) has no route defined at `/` at all — only
`/health` and `/api/tasks` exist — so the health checker's request to
`/` correctly received a 404 from Express's catch-all handler, and the
target group marked it unhealthy, even though the actual app traffic
(`/api/*`, routed by the Ingress rules) worked perfectly.
**Fix:** added `alb.ingress.kubernetes.io/healthcheck-path: /health` to
the Ingress annotations. This one annotation applies to both target
groups the controller manages: it's a genuine, correct 200 on the
backend (the real `/health` route), and on the frontend it still
resolves to a 200 via nginx's SPA fallback (`try_files ... /index.html`)
matching any unrecognized path, `/health` included. **General lesson:**
an app can be fully working end-to-end while its load balancer still
reports it unhealthy — these are two separate checks (real traffic
routing vs. the health check's specific hardcoded path), and a healthy
target group is not automatically implied by the app actually working.
