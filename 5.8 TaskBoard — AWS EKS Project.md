# TaskBoard — Complete EKS Build Runbook

A ground-up, console-first runbook for building a 3-tier application on
Amazon EKS, covering all 13 phases in enough detail to recreate the
entire project from nothing. Built manually through the AWS Console
wherever possible (kubectl/Helm accepted for anything Kubernetes-native
that has no console equivalent), specifically to understand every layer
rather than abstract it away with `eksctl` or Terraform.

**Region:** `ap-south-1` (Mumbai) · **Account ID:** `397483229147` ·
**Cluster name:** `taskboard-cluster`

## 1. Architecture

<img width="1536" height="1024" alt="EKS deployment" src="https://github.com/user-attachments/assets/fb55c5e3-b242-41e6-865a-33487fdfe985" />


Everything inside the EKS cluster stays on private subnets with no
direct internet exposure. The only entry point is the ALB (Phase 8),
itself only reachable through CloudFront once Phase 9 is done. The NAT
Gateway exists solely so private-subnet resources (nodes, pods) can
reach *out* to the internet (pulling images, calling AWS APIs) — nothing
reaches *in* through it.

## Phase 0 — IAM least-privilege setup

**Goal:** never touch this project as root or with `AdministratorAccess`.

1. **IAM Console → Users → Create user**
   - Name: `taskboard-eks-admin`
   - Check "Provide user access to the AWS Management Console"
   - Choose "I want to create an IAM user"
   - Set a custom password, require reset at next sign-in
   - Do **not** attach a policy yet — click through to create the user first

2. **Create the scoped policy — IAM → Policies → Create policy → JSON tab.**
   Paste a policy scoped to exactly this project's needs: `eks:*`; EC2
   networking/compute actions (VPC, subnet, IGW, NAT, route table,
   security group, EC2 instance/launch template, tagging including
   `DeleteTags`); `elasticloadbalancing:*`; ECR scoped to
   `arn:aws:ecr:*:*:repository/taskboard-*` plus the account-wide
   `ecr:GetAuthorizationToken`; `rds:*`; CloudWatch Logs/metrics/dashboards;
   Secrets Manager scoped to `arn:aws:secretsmanager:*:*:secret:taskboard-*`;
   ACM + CloudFront; IAM role create/attach/tag actions scoped to
   `arn:aws:iam::*:role/taskboard-*`, plus a separate `iam:PassRole`
   statement scoped to the same resource pattern with a
   `iam:PassedToService` condition restricted to
   `eks.amazonaws.com`, `ec2.amazonaws.com`, `eks-fargate-pods.amazonaws.com`.
   Name it `TaskBoard-EKS-Project-Policy`.
   **Naming rule for the rest of the project: every IAM role created
   must start with `taskboard-`**, since the policy is scoped to that
   prefix.

3. **Attach the policy** — IAM → Users → `taskboard-eks-admin` → Add
   permissions → Attach policies directly → `TaskBoard-EKS-Project-Policy`.

4. **Enable MFA** — user's Security credentials tab → Assign MFA device →
   Authenticator app → scan QR, confirm two consecutive codes.

5. **Generate access keys for CLI/kubectl/Docker** — same tab → Access
   keys → Create access key → use case "Command Line Interface (CLI)" →
   copy both values immediately (secret key is shown once).

6. **Configure locally:**
   ```bash
   aws configure
   # Access Key ID, Secret Access Key, region: ap-south-1, output: json
   aws sts get-caller-identity   # confirm ARN is taskboard-eks-admin, not root
   ```

7. **Sign in to the console as this user going forward** — never root,
   never a broader admin account, for the rest of this runbook.

## Phase 1 — Networking foundation

**Goal:** a custom VPC with correctly tagged public/private subnets.

1. **VPC Console → Your VPCs → Create VPC**
   - "VPC only" (not the wizard)
   - Name: `taskboard-vpc`, IPv4 CIDR: `10.0.0.0/16`, no IPv6, default tenancy
   - After creation: select it → Actions → Edit VPC settings → confirm
     both **Enable DNS resolution** and **Enable DNS hostnames** are checked
     (required for EKS)

2. **Create 4 subnets** — VPC Console → Subnets → Create subnet, all
   under `taskboard-vpc`:

   | Name | AZ | CIDR |
   |---|---|---|
   | `taskboard-public-1a` | ap-south-1a | `10.0.0.0/24` |
   | `taskboard-public-1b` | ap-south-1b | `10.0.1.0/24` |
   | `taskboard-private-1a` | ap-south-1a | `10.0.10.0/24` |
   | `taskboard-private-1b` | ap-south-1b | `10.0.11.0/24` |

3. **Enable auto-assign public IPv4** on both public subnets only
   (subnet → Actions → Edit subnet settings). Leave unchecked on both
   private subnets.

4. **Internet Gateway** — Create `taskboard-igw`, then Actions → Attach
   to VPC → `taskboard-vpc`.

5. **NAT Gateway** — Create `taskboard-nat` in `taskboard-public-1a`,
   connectivity type Public, allocate a new Elastic IP in the same step.

6. **Route tables:**
   - `taskboard-public-rt` → route `0.0.0.0/0` → `taskboard-igw` →
     associate both public subnets
   - `taskboard-private-rt` → route `0.0.0.0/0` → `taskboard-nat` →
     associate both private subnets

7. **Tag subnets for EKS — critical, do not skip:**

   Public subnets (`taskboard-public-1a`, `taskboard-public-1b`):
   ```
   kubernetes.io/cluster/taskboard-cluster = shared
   kubernetes.io/role/elb = 1
   ```
   Private subnets (`taskboard-private-1a`, `taskboard-private-1b`):
   ```
   kubernetes.io/cluster/taskboard-cluster = shared
   kubernetes.io/role/internal-elb = 1
   ```

8. **Sanity check** — VPC Console → your VPCs → `taskboard-vpc` →
   Resource map tab. Confirm 4 subnets correctly split, IGW attached,
   NAT sitting in a public subnet, both route tables associated correctly.

## Phase 2 — Cluster creation

1. **Cluster IAM role** — IAM → Roles → Create role → AWS service → use
   case **EKS - Cluster** → attaches `AmazonEKSClusterPolicy`
   automatically → name `taskboard-cluster-role`.

2. **Create the cluster** — EKS Console → Clusters → Create cluster:
   - Name: `taskboard-cluster`, latest stable Kubernetes version
   - Cluster service role: `taskboard-cluster-role`
   - Networking → VPC: `taskboard-vpc`; **Subnets: private subnets only**
     (`taskboard-private-1a`, `taskboard-private-1b`) — deliberately
     excludes public subnets; the ALB Controller (Phase 8) discovers
     subnets by VPC-wide tag, not by cluster association, so this
     doesn't block anything later
   - Cluster endpoint access: **Public and private** (public so you can
     kubectl from your laptop; private so node↔control-plane traffic
     stays inside the VPC)
   - Control plane egress: **AWS managed** (Customer routed is only
     needed for Hybrid Nodes / on-prem, not applicable here, and it's a
     one-way migration so leave the default)
   - Metrics: leave **Prometheus** and **CloudWatch** both **off** here —
     both get installed manually and deliberately in Phase 11 instead
   - Container network observability: **off**
   - Control plane logging: enable **all 5** — API server, Audit,
     Authenticator, Controller manager, Scheduler
   - Add-ons screen: select only **CoreDNS**, **kube-proxy**,
     **Amazon VPC CNI** — leave EBS CSI, EFS CSI, CloudWatch Observability,
     Load Balancer Controller, S3 CSI, node monitoring agent, GuardDuty
     agent, and Pod Identity Agent all unchecked (each gets added
     deliberately, later, with a manually-built IRSA role)
   - Create, wait ~10-15 minutes for **Active**

3. **Connect kubectl:**
   ```bash
   aws eks update-kubeconfig --region ap-south-1 --name taskboard-cluster
   kubectl get svc   # should show the default kubernetes ClusterIP service
   ```

4. **Create the OIDC provider manually** (this is the step that most
   often gets skipped since it's not automatic):
   ```bash
   aws eks describe-cluster --name taskboard-cluster --region ap-south-1 \
     --query "cluster.identity.oidc.issuer" --output text
   ```
   Copy the URL, then **IAM → Identity providers → Add provider →
   OpenID Connect** → paste the URL → **Get thumbprint** → Audience:
   `sts.amazonaws.com` → Add provider.

   Verify:
   ```bash
   aws iam list-open-id-connect-providers
   aws iam get-open-id-connect-provider --open-id-connect-provider-arn \
     arn:aws:iam::397483229147:oidc-provider/oidc.eks.ap-south-1.amazonaws.com/id/<oidc-id>
   # confirm "ClientIDList": ["sts.amazonaws.com"]
   ```

**Why the OIDC provider matters:** EKS auto-creates an OIDC *issuer*
(capable of signing tokens) the moment the cluster exists. But IAM won't
trust tokens from it until you separately register it as an IAM OIDC
*provider* — that's this step. Without it, IRSA (Phase 5 onward) cannot
work at all.

## Phase 3 — Compute

1. **Node IAM role** — IAM → Roles → Create role → AWS service → EC2 →
   attach `AmazonEKSWorkerNodePolicy`, `AmazonEKS_CNI_Policy`,
   `AmazonEC2ContainerRegistryReadOnly` → name `taskboard-node-role`.

2. **Managed node group** — EKS Console → taskboard-cluster → Compute →
   Add node group:
   - Name: `taskboard-nodegroup`, IAM role: `taskboard-node-role`
   - AMI: Amazon Linux 2023 (x86_64), instance type `t3.medium`, disk 20 GiB
   - Scaling: desired 2, min 2, max 4
   - Update config: max unavailable = **1** (number, not percentage),
     update strategy **Default** (not Minimal)
   - Node auto repair: **enabled**
   - Subnets: private subnets only
   - Remote access (SSH): **disabled**
   - Kubernetes labels: none. Kubernetes taints: none (this is the
     general-purpose node group; a tainted Spot node group is the bonus
     exercise at the end of this document)
   - Create, wait ~3-5 minutes

3. **Fargate profile** — Compute tab → Add Fargate profile:
   - Name: `taskboard-fargate-profile`
   - Pod execution role: let the console create a new one (this is one
     of the few cases where console-generated is fine — Fargate's trust
     policy for `eks-fargate-pods.amazonaws.com` is fixed and specific)
   - Subnets: private only (Fargate doesn't support public subnets at all)
   - Namespace: `fargate-workloads`
   - Create

4. **Create the namespace and verify:**
   ```bash
   kubectl create namespace fargate-workloads
   kubectl get nodes                      # 2 nodes, Ready, IPs in 10.0.10.x/10.0.11.x
   kubectl get namespace fargate-workloads
   ```

## Phase 4 — Container images and ECR

**The app itself:** a 3-tier kanban board — React (Vite) frontend, Node/
Express backend with a `pg` (node-postgres) client, Postgres as the data
tier. Backend exposes `/api/tasks` (GET/POST/PATCH/DELETE) and `/health`
(deliberately DB-independent, used later as the liveness/readiness probe
target). Full source: `backend/src/{index,db,routes/tasks}.js`,
`frontend/src/{App,api,index.css,components/*}.jsx`.

**Dockerfiles:**
- **Backend**: single-stage `node:20-alpine`, `COPY package.json` +
  `npm install --omit=dev` *before* `COPY src` (maximizes layer cache
  reuse — code changes don't force a dependency reinstall), runs as a
  non-root user, exec-form `CMD ["node", "src/index.js"]` (ensures
  SIGTERM reaches the process directly for clean shutdowns).
- **Frontend**: multi-stage — Stage 1 (`node:20-alpine AS build`) runs
  `npm install` + `npm run build`, producing a `dist/` folder. Stage 2
  (`nginx:1.27-alpine`) does `COPY --from=build /app/dist ...` — pulling
  forward *only* the built static files, discarding `node_modules`,
  source, and all build tooling entirely. Final image ships nginx +
  static files only, ~40MB, no Node.js runtime at all.

1. **Create ECR repos** — ECR Console → Repositories → Create repository
   (Private) → `taskboard-frontend`, then again for `taskboard-backend`.
   Scan on push: optional but free, worth enabling.

2. **Authenticate Docker to ECR:**
   ```bash
   export AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
   export AWS_REGION=ap-south-1
   export ECR_REGISTRY=$AWS_ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com
   aws ecr get-login-password --region $AWS_REGION | \
     docker login --username AWS --password-stdin $ECR_REGISTRY
   ```

3. **Build and push — `--platform linux/amd64` is mandatory on Apple
   Silicon** (M1/M2 Macs build arm64 by default; the node group's EC2
   instances are x86_64 — without this flag the image pulls fine but
   crash-loops with an exec format error):
   ```bash
   docker build --platform linux/amd64 -t $ECR_REGISTRY/taskboard-backend:v1 ./backend
   docker push $ECR_REGISTRY/taskboard-backend:v1

   docker build --platform linux/amd64 -t $ECR_REGISTRY/taskboard-frontend:v1 ./frontend
   docker push $ECR_REGISTRY/taskboard-frontend:v1
   ```

4. **Verify:**
   ```bash
   aws ecr describe-images --repository-name taskboard-backend --region ap-south-1 \
     --query "imageDetails[].imageTags"
   aws ecr describe-images --repository-name taskboard-frontend --region ap-south-1 \
     --query "imageDetails[].imageTags"
   ```

**Test locally with docker-compose before ever touching ECR** — a
`docker-compose.yml` spinning up `postgres:16-alpine` + backend + frontend
together lets you verify the whole stack works in seconds, for free, before
any AWS resource exists. `docker compose up --build`, open
`http://localhost:8080`.

## Phase 5 — Add-ons and IRSA

1. **Simple add-ons** — EKS Console → taskboard-cluster → Add-ons →
   Get more add-ons → select **CoreDNS**, **kube-proxy**,
   **Amazon VPC CNI** together, default versions, Create. None need IAM.
   ```bash
   kubectl get pods -n kube-system   # coredns, aws-node, kube-proxy all Running
   ```

2. **Get OIDC details** (needed for every IRSA role from here on):
   - Account ID: shown in the console's top-right account menu
   - OIDC issuer ID: EKS Console → taskboard-cluster → Overview tab →
     "OpenID Connect provider URL" → the ID after `/id/`

3. **Create the IAM role for EBS CSI — Web Identity method:**
   - IAM → Roles → Create role → Trusted entity type: **Web identity**
   - Identity provider: select your cluster's OIDC provider from the dropdown
   - Audience: `sts.amazonaws.com`
   - Permissions: `AmazonEBSCSIDriverPolicy`
   - Name: `taskboard-ebs-csi-role`

4. **Tighten the trust policy to the exact service account** — this is
   the actual security-relevant IRSA mechanism:
   - Role → Trust relationships → Edit trust policy → ensure the
     `Condition` block reads:
   ```json
   "Condition": {
     "StringEquals": {
       "oidc.eks.ap-south-1.amazonaws.com/id/<oidc-id>:aud": "sts.amazonaws.com",
       "oidc.eks.ap-south-1.amazonaws.com/id/<oidc-id>:sub": "system:serviceaccount:kube-system:ebs-csi-controller-sa"
     }
   }
   ```
   The `:sub` condition is what restricts this role to *only* be
   assumable by that exact service account — without it, any pod with
   any service account and a validly-signed OIDC token could assume it.

5. **Install the add-on, selecting the role during creation (not after)** —
   Add-ons → Get more add-ons → Amazon EBS CSI Driver → Next → **expand
   "Optional configuration settings" explicitly** → IAM role:
   `taskboard-ebs-csi-role` → Next → Create.

6. **Repeat steps 3-5 for EFS CSI**, using `AmazonEFSCSIDriverPolicy` and
   `:sub` = `system:serviceaccount:kube-system:efs-csi-controller-sa`,
   role name `taskboard-efs-csi-role`.

7. **Verify both:**
   ```bash
   kubectl get pods -n kube-system | grep -E 'ebs-csi|efs-csi'   # all Running, full readiness
   kubectl describe sa ebs-csi-controller-sa -n kube-system
   kubectl describe sa efs-csi-controller-sa -n kube-system
   # both should show: Annotations: eks.amazonaws.com/role-arn: arn:aws:iam::...:role/taskboard-...-csi-role
   ```

   **Known console reliability issue:** the add-on wizard sometimes
   fails to persist the IAM role selection even when it was visibly
   selected — service accounts show `Annotations: <none>` and controller
   pods crash-loop with partial readiness (e.g. `1/6`). If this happens:
   remove and recreate the add-on, watching the role dropdown carefully;
   if it happens again, set the annotation directly and force a restart:
   ```bash
   kubectl annotate serviceaccount ebs-csi-controller-sa -n kube-system \
     eks.amazonaws.com/role-arn=arn:aws:iam::397483229147:role/taskboard-ebs-csi-role --overwrite
   kubectl delete pod -n kube-system -l app.kubernetes.io/name=aws-ebs-csi-driver
   ```
   (Same pattern for EFS, substituting the EFS role/label.) See the
   [Bugs](#bugs-encountered-and-fixes) section for the full writeup.

## Phase 6 — Deploy the app

1. **DB subnet group** — RDS Console → Subnet groups → Create →
   `taskboard-db-subnet-group`, VPC `taskboard-vpc`, both private subnets.

2. **RDS security group** — VPC Console → Security groups → Create →
   `taskboard-rds-sg`, VPC `taskboard-vpc`. Inbound: PostgreSQL (5432),
   **source = your node group's security group** (EKS Console →
   taskboard-cluster → Compute → node group → Details — usually named
   `eks-cluster-sg-taskboard-cluster-...`). This is the actual mechanism
   that lets pods reach RDS — pods inherit node security groups via the
   VPC CNI; it isn't about subnet proximity.

3. **Create the RDS instance** — RDS Console → Create database:
   - Engine PostgreSQL, latest stable, template Free tier/Dev-Test
   - Identifier `taskboard-db`, master username `taskboard`, set a password
   - Instance class `db.t3.micro`, 20 GiB gp3, Multi-AZ **No**
   - VPC `taskboard-vpc`, DB subnet group `taskboard-db-subnet-group`
   - Public access **No**, security group → existing → `taskboard-rds-sg`
   - **Initial database name: `taskboard`** — do not leave this blank;
     skipping it means only the default `postgres` database exists and
     your app will fail with `database "taskboard" does not exist`
   - Create, wait 5-10 min, copy the endpoint once Available

4. **Namespace and Secret:**
   ```bash
   kubectl create namespace taskboard
   kubectl create secret generic taskboard-db-secret \
     --namespace taskboard --from-literal=PGPASSWORD='<your-rds-password>'
   ```

5. **Backend Deployment + Service** (`backend-deployment.yaml`):
   ```yaml
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: taskboard-backend
     namespace: taskboard
   spec:
     replicas: 2
     selector:
       matchLabels:
         app: taskboard-backend
     template:
       metadata:
         labels:
           app: taskboard-backend
       spec:
         containers:
           - name: backend
             image: 397483229147.dkr.ecr.ap-south-1.amazonaws.com/taskboard-backend:v2
             ports:
               - containerPort: 4000
             env:
               - name: PGHOST
                 value: "<your-rds-endpoint>"
               - name: PGPORT
                 value: "5432"
               - name: PGUSER
                 value: "taskboard"
               - name: PGDATABASE
                 value: "taskboard"
               - name: PORT
                 value: "4000"
               - name: PGSSL
                 value: "true"
               - name: PGPASSWORD
                 valueFrom:
                   secretKeyRef:
                     name: taskboard-db-secret
                     key: PGPASSWORD
             readinessProbe:
               httpGet: { path: /health, port: 4000 }
               initialDelaySeconds: 5
               periodSeconds: 10
             livenessProbe:
               httpGet: { path: /health, port: 4000 }
               initialDelaySeconds: 15
               periodSeconds: 20
             resources:
               requests: { cpu: "100m", memory: "128Mi" }
               limits: { cpu: "300m", memory: "256Mi" }
   ---
   apiVersion: v1
   kind: Service
   metadata:
     name: taskboard-backend-svc
     namespace: taskboard
   spec:
     selector:
       app: taskboard-backend
     ports:
       - port: 4000
         targetPort: 4000
   ```
   (`PGSSL: "true"` and image tag `:v2` reflect a fix from the first
   build — see Bugs section. `readinessProbe` controls whether traffic
   routes to the pod; `livenessProbe` controls whether Kubernetes
   restarts it. Both hit `/health`, which never touches the DB — a slow
   RDS shouldn't cause Kubernetes to think the *pod* is broken.)

6. **Frontend Deployment + Service** (`frontend-deployment.yaml`) — same
   shape, no `env` block, no probes, `containerPort: 80`,
   `image: .../taskboard-frontend:v3` (tag reflects the nginx DNS fix,
   see Bugs), Service `port: 80`/`targetPort: 80`. Both Services default
   to `ClusterIP` (no `type:` field) — deliberately not internet-facing
   yet; Phase 8's ALB Ingress is the intended external entry point.

7. **Apply and verify:**
   ```bash
   kubectl apply -f backend-deployment.yaml
   kubectl apply -f frontend-deployment.yaml
   kubectl get pods -n taskboard
   kubectl logs deployment/taskboard-backend -n taskboard   # look for "Database schema ready"
   kubectl port-forward svc/taskboard-frontend-svc -n taskboard 8080:80
   # open http://localhost:8080, add a task, confirm it persists
   ```

## Phase 7 — Storage

**EBS (per-pod local storage, `ReadWriteOnce`):**

1. Create a StorageClass (no console equivalent — this is a pure
   Kubernetes object):
   ```yaml
   apiVersion: storage.k8s.io/v1
   kind: StorageClass
   metadata:
     name: taskboard-gp3
   provisioner: ebs.csi.aws.com
   volumeBindingMode: WaitForFirstConsumer
   parameters:
     type: gp3
   ```
2. Create a PVC referencing it:
   ```yaml
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata:
     name: taskboard-backend-data
     namespace: taskboard
   spec:
     accessModes: ["ReadWriteOnce"]
     storageClassName: taskboard-gp3
     resources:
       requests:
         storage: 5Gi
   ```
3. Mount it into the backend Deployment (add under the container spec):
   ```yaml
             volumeMounts:
               - name: backend-data
                 mountPath: /app/data
         volumes:
           - name: backend-data
             persistentVolumeClaim:
               claimName: taskboard-backend-data
   ```
4. Apply, then verify: `kubectl get pvc,pv -n taskboard` (should show
   `Bound`), and check the EC2 Console → Volumes for a new gp3 volume
   tagged with the PVC name — confirms the EBS CSI IRSA role from
   Phase 5 is actually doing real provisioning work.

**EFS (shared storage across pods, `ReadWriteMany`):**

1. **EFS Console → Create file system** → VPC `taskboard-vpc` → mount
   targets in both private subnets.
2. **Security group for EFS** — same pattern as RDS: create
   `taskboard-efs-sg`, inbound NFS (port 2049) from the node group's
   security group; attach it to the EFS mount targets.
3. **StorageClass for dynamic provisioning:**
   ```yaml
   apiVersion: storage.k8s.io/v1
   kind: StorageClass
   metadata:
     name: taskboard-efs-sc
   provisioner: efs.csi.aws.com
   parameters:
     provisioningMode: efs-ap
     fileSystemId: fs-XXXXXXXX   # from the EFS console
     directoryPerms: "700"
   ```
4. **PVC:**
   ```yaml
   apiVersion: v1
   kind: PersistentVolumeClaim
   metadata:
     name: taskboard-shared-storage
     namespace: taskboard
   spec:
     accessModes: ["ReadWriteMany"]
     storageClassName: taskboard-efs-sc
     resources:
       requests:
         storage: 5Gi
   ```
5. Mount at e.g. `/app/shared` in the backend Deployment (same
   `volumeMounts`/`volumes` pattern as EBS above).
6. **Verify `ReadWriteMany` actually works across pods** — exec into one
   backend replica, write a file into `/app/shared`, exec into the
   *other* replica, confirm the file is visible there too. That's the
   proof this isn't behaving like EBS.

## Phase 8 — Ingress and load balancing

1. **Download AWS's official IAM policy for the Load Balancer Controller**
   (from the `aws-load-balancer-controller` GitHub repo's
   `iam_policy.json`) and create it manually: IAM → Policies → Create
   policy → JSON → paste it → name `taskboard-alb-controller-policy`.

2. **IRSA role, same Web Identity pattern as Phase 5:**
   - Trust type Web identity, your OIDC provider, audience `sts.amazonaws.com`
   - Attach `taskboard-alb-controller-policy`
   - Name `taskboard-alb-controller-role`
   - Trust policy `:sub` = `system:serviceaccount:kube-system:aws-load-balancer-controller`

3. **Install via Helm** (no console equivalent for this controller):
   ```bash
   helm repo add eks https://aws.github.io/eks-charts
   helm repo update
   helm install aws-load-balancer-controller eks/aws-load-balancer-controller \
     -n kube-system \
     --set clusterName=taskboard-cluster \
     --set region=ap-south-1 \
     --set vpcId=<your-vpc-id> \
     --set serviceAccount.create=true \
     --set serviceAccount.name=aws-load-balancer-controller \
     --set serviceAccount.annotations."eks\.amazonaws\.com/role-arn"=arn:aws:iam::397483229147:role/taskboard-alb-controller-role
   ```
   Verify: `kubectl get deployment -n kube-system aws-load-balancer-controller`
   should show it Running.

4. **Create the Ingress** (`taskboard-ingress.yaml`):
   ```yaml
   apiVersion: networking.k8s.io/v1
   kind: Ingress
   metadata:
     name: taskboard-ingress
     namespace: taskboard
     annotations:
       kubernetes.io/ingress.class: alb
       alb.ingress.kubernetes.io/scheme: internet-facing
       alb.ingress.kubernetes.io/target-type: ip
   spec:
     rules:
       - http:
           paths:
             - path: /api
               pathType: Prefix
               backend:
                 service: { name: taskboard-backend-svc, port: { number: 4000 } }
             - path: /
               pathType: Prefix
               backend:
                 service: { name: taskboard-frontend-svc, port: { number: 80 } }
   ```
   Order matters — `/api` must come before the catch-all `/`, since
   rules are evaluated top to bottom. `target-type: ip` routes straight
   to pod IPs (works because of the VPC CNI). This is also the point
   where the frontend's `nginx.conf` `/api/` proxy block becomes
   genuinely unused in production — the ALB intercepts `/api/*` before
   it ever reaches the frontend pod.

5. **Apply and verify:**
   ```bash
   kubectl apply -f taskboard-ingress.yaml
   kubectl get ingress -n taskboard   # ADDRESS column = the ALB's DNS name, once provisioned (few minutes)
   curl http://<alb-dns-name>/            # frontend
   curl http://<alb-dns-name>/api/health  # backend
   ```

## Phase 9 — CDN and custom domain

1. **Request an ACM certificate in `us-east-1`** — CloudFront requires
   its certificate to live in this specific region regardless of where
   your cluster is. ACM Console (switch region to N. Virginia) → Request
   certificate → Public → your domain (e.g. `taskboard.yourdomain.com`)
   → DNS validation.

2. **Add the validation CNAME at your own DNS provider** — ACM shows a
   CNAME name/value pair; add it there. Wait for status **Issued**
   (can take a few minutes to a few hours depending on your DNS provider's TTL).

3. **Create the CloudFront distribution** — CloudFront Console → Create
   distribution:
   - Origin domain: your ALB's DNS name (from Phase 8)
   - Origin protocol policy: HTTP only (simplification — the ALB has no
     HTTPS listener in this project; adding one would need a *regional*
     ACM cert in `ap-south-1`, separate from this `us-east-1` one, as a
     Phase 12+ hardening exercise)
   - Viewer protocol policy: Redirect HTTP to HTTPS
   - Cache behavior for `/api/*`: cache disabled (or TTL ~0), forward all
     headers/cookies — API responses shouldn't be cached like static assets
   - Default cache behavior (`/*`): standard caching enabled
   - Alternate domain name (CNAME): your custom domain
   - Custom SSL certificate: the ACM cert from step 1

4. **Point your own DNS at CloudFront** — add a CNAME (or ALIAS/ANAME if
   your provider supports it for a root domain) from your domain to the
   distribution's `*.cloudfront.net` address.

5. **Verify:**
   ```bash
   dig yourdomain.com   # or nslookup
   curl -I https://yourdomain.com
   # look for an X-Cache header confirming CloudFront is actually in the path
   ```

## Phase 10 — Autoscaling

**Cluster Autoscaler:**

1. Tag the node group's underlying Auto Scaling Group (EC2 Console →
   Auto Scaling Groups → find the one tied to `taskboard-nodegroup` →
   Tags):
   ```
   k8s.io/cluster-autoscaler/enabled = true
   k8s.io/cluster-autoscaler/taskboard-cluster = owned
   ```
2. Create a custom IAM policy covering the standard Cluster Autoscaler
   actions (`autoscaling:DescribeAutoScalingGroups`,
   `autoscaling:DescribeAutoScalingInstances`,
   `autoscaling:DescribeLaunchConfigurations`, `autoscaling:DescribeTags`,
   `autoscaling:SetDesiredCapacity`,
   `autoscaling:TerminateInstanceInAutoScalingGroup`,
   `ec2:DescribeLaunchTemplateVersions`) → name
   `taskboard-cluster-autoscaler-policy`.
3. IRSA role `taskboard-cluster-autoscaler-role`, same Web Identity
   pattern, `:sub` = `system:serviceaccount:kube-system:cluster-autoscaler`.
4. Deploy via the official `cluster-autoscaler-autodiscover.yaml`
   manifest (from the Kubernetes autoscaler GitHub repo), editing the
   `--node-group-auto-discovery` flag to reference
   `taskboard-cluster`, and annotating its ServiceAccount with the IRSA role ARN.
5. **Verify** by scaling a deployment past current node capacity
   (`kubectl scale deployment taskboard-backend -n taskboard --replicas=20`)
   and watching `kubectl get nodes -w` — new nodes should appear as the
   autoscaler reacts.

**HPA (Horizontal Pod Autoscaler):**

1. Install **metrics-server** first — HPA has no data to scale on
   without it (not included in EKS by default): apply the official
   `components.yaml` from the `kubernetes-sigs/metrics-server` repo.
2. Create the HPA:
   ```bash
   kubectl autoscale deployment taskboard-backend -n taskboard \
     --cpu-percent=60 --min=2 --max=6
   ```
3. **Verify** with `kubectl get hpa -n taskboard -w` while generating
   load against the backend (a simple loop of `curl` against
   `/api/tasks`, or a small load-testing pod) — watch `REPLICAS` climb.

## Phase 12 — Security hardening

**NetworkPolicy:**
The VPC CNI supports native NetworkPolicy enforcement (or install
Calico as an alternative policy engine). Example — restrict the backend
to only accept traffic from the frontend pods and deny everything else:
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-allow-frontend-only
  namespace: taskboard
spec:
  podSelector:
    matchLabels:
      app: taskboard-backend
  policyTypes: ["Ingress"]
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: taskboard-frontend
      ports:
        - port: 4000
```

**Secrets Manager instead of a plain Kubernetes Secret:**
1. Store the real secret: Secrets Manager Console → Store a new secret
   → Credentials for RDS database → reference `taskboard-db`.
2. Install the Secrets Store CSI Driver (Helm chart
   `secrets-store-csi-driver/secrets-store-csi-driver`) plus the AWS
   provider (manifest from the
   `aws-secrets-manager-and-config-provider-for-secrets-store-csi-driver`
   GitHub repo).
3. IRSA role `taskboard-secrets-csi-role`, policy scoped to
   `secretsmanager:GetSecretValue` on that one secret's ARN only.
4. Create a `SecretProviderClass` referencing the secret, mount it as a
   volume in the backend Deployment, and use its `secretObjects` field
   to sync it into a real Kubernetes Secret that `PGPASSWORD`'s
   `secretKeyRef` then points to — replacing the manually-created
   `taskboard-db-secret` from Phase 6 with something backed by rotation-
   capable Secrets Manager instead of a static value.

## Phase 13 — Cleanup

Tear down in dependency order (reverse of build order), confirming each
step before moving to the next:

1. CloudFront — disable the distribution first, wait for it to finish
   deploying disabled, then delete it. Delete the ACM cert afterward.
2. `kubectl delete ingress taskboard-ingress -n taskboard` — the ALB
   Controller removes the actual ALB automatically.
3. `helm uninstall` for: kube-prometheus-stack, aws-load-balancer-controller,
   cluster-autoscaler (if installed via Helm), secrets-store-csi-driver.
4. Delete PVCs (`kubectl delete pvc --all -n taskboard`) — confirm the
   underlying EBS volumes and EFS access points are actually released in
   their respective consoles afterward; delete the EFS file system itself.
5. Delete the managed node group, then the Fargate profile.
6. Delete the EKS cluster.
7. Delete the OIDC identity provider (IAM → Identity providers).
8. Delete the RDS instance (disable deletion protection first if enabled;
   skip the final snapshot if the data doesn't need preserving).
9. Delete the NAT Gateway, then release its Elastic IP.
10. Delete route tables, detach and delete the Internet Gateway, delete
    subnets, delete the VPC.
11. Optional, only if fully done with the project: IAM roles/policies/
    user, ECR repositories (keep these if you want the images preserved).

## Bonus — Spot node group with taints/tolerations

A hands-on exercise for after Phase 13, or any time once the core
project works, to see taints/tolerations in action rather than just
read about them:

1. **Create a second managed node group** — `taskboard-spot-nodegroup`,
   capacity type **Spot**, a diversified instance type list (e.g.
   `t3.medium`, `t3a.medium`, `t2.medium` — Spot availability improves
   with more instance type options), private subnets, same node IAM role.
2. **Add a Kubernetes taint during creation** (this time, actually fill
   in the taints section skipped in Phase 3):
   ```
   key: spot   value: true   effect: NoSchedule
   ```
3. **Deploy a test pod with the matching toleration:**
   ```yaml
   apiVersion: v1
   kind: Pod
   metadata:
     name: spot-tolerant-test
     namespace: taskboard
   spec:
     tolerations:
       - key: "spot"
         operator: "Equal"
         value: "true"
         effect: "NoSchedule"
     containers:
       - name: busybox
         image: busybox
         command: ["sleep", "3600"]
   ```
4. **Verify it lands on the Spot node group:**
   ```bash
   kubectl get pod spot-tolerant-test -n taskboard -o wide   # check the NODE column
   ```
5. **Prove the taint actually repels untolerated pods** — deploy an
   identical pod *without* the `tolerations` block, and confirm via
   `kubectl get pod -o wide` that it schedules onto the regular
   `taskboard-nodegroup` instead, never landing on the tainted Spot nodes.

## Bugs encountered and fixes

### 1. RDS SSL enforcement rejecting the backend's connection
**Symptom:** `no pg_hba.conf entry for host "10.0.10.5"... no encryption`
**Cause:** RDS Postgres enforces SSL by default; the `pg` library
defaults to no SSL. Never surfaced locally since docker-compose Postgres
doesn't enforce SSL at all.
**Fix:** `ssl: process.env.PGSSL === 'true' ? { rejectUnauthorized: false } : false`
in `db.js`'s `Pool` config; set `PGSSL=true` only in the Kubernetes
Deployment env, leaving local docker-compose untouched. Rebuilt as `:v2`.
*(Phase 12+ hardening: validate against the real RDS CA bundle instead
of `rejectUnauthorized: false`.)*

### 2. Target database never created inside the RDS instance
**Symptom:** `database "taskboard" does not exist`
**Cause:** "Initial database name" left blank during RDS creation — the
master **user** existed, the **database** did not.
**Fix:** Temporary self-deleting pod to run SQL from inside the VPC
(RDS is private-only, unreachable from a laptop directly):
```bash
kubectl run psql-client --rm -it --restart=Never -n taskboard \
  --image=postgres:16-alpine -- \
  psql "postgresql://taskboard:<password>@<rds-endpoint>:5432/postgres?sslmode=require"
```
then `CREATE DATABASE taskboard OWNER taskboard;`. This pod could reach
RDS with zero extra security-group changes because pods inherit their
node's security groups via the VPC CNI, and the node group's SG was
already allow-listed on the RDS SG. **Prevention:** always fill in
"Initial database name" during RDS creation (see Phase 6, step 3).

### 3. nginx crash-looping / 502 due to a docker-compose-only hostname
**Symptom (boot crash):** `nginx: [emerg] host not found in upstream "backend"`
**Symptom (post-partial-fix, on use):** `502 Bad Gateway` when adding a task
**Cause:** `nginx.conf`'s `/api/` proxy block pointed at `backend:4000` —
a hostname only meaningful inside docker-compose's private network.
Static `proxy_pass` targets resolve once at nginx startup; failed
resolution crashes nginx entirely. In Kubernetes, `backend` doesn't
resolve to anything.
**Fix (two-part):** (1) switched to a variable-based `proxy_pass`
(`set $upstream_backend ...;`) so resolution happens per-request instead
of at boot, stopping the crash; (2) pointed the variable at the real
Kubernetes Service DNS name — `taskboard-backend-svc.taskboard.svc.cluster.local:4000`
— with an explicit `resolver` directive using CoreDNS's ClusterIP
(`kubectl get svc kube-dns -n kube-system`). Rebuilt as `:v3`. Note: this
block only matters for `kubectl port-forward` testing — Phase 8's ALB
intercepts `/api/*` before it reaches the frontend pod in production.

### 4. EKS add-on console UI not persisting the IRSA role selection
**Symptom:** service account shows `Annotations: <none>` after selecting
an IAM role during EBS/EFS CSI Driver add-on setup; controller pods
crash-loop with partial readiness (e.g. `1/6`) and credential errors.
Recurred on both drivers, at both the original setup and again after a
full cluster rebuild.
**Root cause:** apparent console reliability issue — the add-on wizard
silently fails to write the `eks.amazonaws.com/role-arn` annotation even
when the role was visibly selected pre-Create.
**Fix:** set the annotation directly and force pod recreation:
```bash
kubectl annotate serviceaccount ebs-csi-controller-sa -n kube-system \
  eks.amazonaws.com/role-arn=arn:aws:iam::397483229147:role/taskboard-ebs-csi-role --overwrite
kubectl delete pod -n kube-system -l app.kubernetes.io/name=aws-ebs-csi-driver
```
(mirror for EFS). Ruled out along the way: trust policy correctness
(confirmed correct via IAM JSON view both times) and node schedulability
(`InsufficientNumberOfReplicas` briefly appeared mid-rebuild simply
because the node group hadn't finished initializing — resolved once
`kubectl get nodes` showed `Ready`, no action needed).

### 5. Cost-driven full EKS teardown and rebuild
Not a bug — a deliberate exercise. EKS control plane has no stop/pause
option (delete-or-nothing), so it was fully deleted (Fargate profile →
node group → cluster → OIDC provider, in that order) to stop billing
overnight. **RDS was stopped, not deleted** — RDS supports up to a 7-day
stop, preserving data, master user, and endpoint hostname, at far less
effort than deletion. Rebuilding EKS required: a new cluster, a **new**
OIDC provider (new clusters mint a new OIDC ID), **updating both CSI
IRSA roles' trust policies** with the new ID, recreating the node group
and Fargate profile, and re-hitting (and re-fixing, via the same kubectl
annotate workaround) bug #4 on both CSI drivers. Because RDS was only
stopped, no database/security-group/`PGHOST` changes were needed there.
