# Self Healing 3-Tier Web App on AWS — Manual Build Guide

A step-by-step POC covering three layers of self-healing: infrastructure (EC2 + Auto Scaling), container orchestration (ECS Fargate), and automated remediation (CloudWatch → EventBridge → Lambda → SSM), validated with AWS Fault Injection Simulator (FIS).

All steps use the AWS Console. CLI equivalents are included where they save time. Estimated build time: 3–5 hours. Estimated cost: a few dollars/day if left running — see Cleanup at the end.

---

## Prerequisites

- An AWS account with admin-level access (or a role with EC2, ECS, Lambda, CloudWatch, EventBridge, SSM, FIS, and IAM permissions)
- AWS CLI installed and configured (`aws configure`) — optional but recommended for faster testing
- A region picked and used consistently throughout (e.g., `us-east-1`)
- Default VPC is fine for a POC — no need to build custom networking

---

## Phase 1: Infra-layer self-healing (EC2 + ALB + Auto Scaling Group)

**Goal:** prove that a failed EC2 instance is automatically detected and replaced.

### 1.1 Create a launch template
1. EC2 Console → Launch Templates → Create launch template
2. Name: `selfheal-web-lt`
3. AMI: Ubuntu Server 22.04 LTS (or 24.04 LTS)
4. Instance type: `t3.micro`
5. Key pair: create or select one (SSH default user is `ubuntu`, e.g. `ssh -i key.pem ubuntu@<public-ip>` — only needed if you want to debug on the box)
6. Security group: allow inbound HTTP (80) from anywhere, SSH (22) from your IP
7. Under Advanced details → User data, paste a script that installs a simple web server and a fake "failure" endpoint:

```bash
#!/bin/bash
apt-get update -y
apt-get install -y apache2
systemctl enable apache2
systemctl start apache2
echo "OK - $(hostname -f)" > /var/www/html/index.html
# Fake error endpoint to trigger alarms later
mkdir -p /var/www/html/error
echo "Simulated 500" > /var/www/html/error/index.html
```

### 1.2 Create a target group
1. EC2 Console → Target Groups → Create target group
2. Target type: Instances
3. Protocol: HTTP, Port 80
4. Health check path: `/`
5. Name: `selfheal-tg`

### 1.3 Create an Application Load Balancer
1. EC2 Console → Load Balancers → Create → Application Load Balancer
2. Name: `selfheal-alb`
3. Scheme: internet-facing
4. Listeners: HTTP:80 → forward to `selfheal-tg`
5. Select at least 2 subnets in different AZs

### 1.4 Create the Auto Scaling Group
1. EC2 Console → Auto Scaling Groups → Create
2. Name: `selfheal-asg`
3. Launch template: `selfheal-web-lt`
4. Attach to existing target group: `selfheal-tg`
5. Health checks: enable **ELB health checks** (not just EC2 status checks — this is what makes it detect app-level failure, not just instance failure)
6. Desired capacity: 2, Min: 2, Max: 4

### 1.5 Test it manually
- Wait for instances to pass health checks (target group shows "healthy")
- Terminate one instance manually: EC2 Console → Instances → select one → Terminate
- Watch the Auto Scaling Group activity history — it should launch a replacement within 1–2 minutes
- CLI check:
```bash
aws autoscaling describe-scaling-activities --auto-scaling-group-name selfheal-asg
```

This is your first self-healing loop: **detect (ELB health check fails) → act (ASG terminates and relaunches) → verify (new instance passes health check)**.

## Phase 2: Container-layer self-healing (ECS Fargate)

**Goal:** prove that a killed container task is automatically rescheduled, independent of any host.

### 2.1 Push a container image
1. Create an ECR repository: ECR Console → Create repository → `selfheal-app`
2. Build and push a minimal image (reuse the same nginx/httpd logic, or use a public image like `public.ecr.aws/nginx/nginx:latest` for speed if you don't want to build one)
```bash
aws ecr get-login-password --region <region> | docker login --username AWS --password-stdin <account-id>.dkr.ecr.<region>.amazonaws.com
docker build -t selfheal-app .
docker tag selfheal-app:latest <account-id>.dkr.ecr.<region>.amazonaws.com/selfheal-app:latest
docker push <account-id>.dkr.ecr.<region>.amazonaws.com/selfheal-app:latest
```

### 2.2 Create an ECS cluster
1. ECS Console → Clusters → Create cluster
2. Infrastructure: AWS Fargate
3. Name: `selfheal-cluster`

### 2.3 Create a task definition
1. ECS Console → Task Definitions → Create new task definition
2. Launch type: Fargate
3. Container image: your ECR image URI
4. Port mapping: 80
5. Add a container health check if you want extra realism (e.g., `CMD-SHELL, curl -f http://localhost/ || exit 1`)

### 2.4 Create an ECS service
1. Inside `selfheal-cluster` → Create Service
2. Launch type: Fargate
3. Task definition: the one above
4. Desired tasks: 2
5. Optionally attach a second ALB listener/target group so you can hit this tier separately from the EC2 tier

### 2.5 Test it manually
- ECS Console → Cluster → Service → Tasks tab
- Stop one task manually
- Watch the service automatically start a replacement task to maintain desired count of 2
- CLI check:
```bash
aws ecs describe-services --cluster selfheal-cluster --services selfheal-service
```

This is self-healing at the orchestration layer: ECS's own control loop keeps the running count matched to the desired count, with no ASG or EC2 involvement at all.

## Phase 3: Automated remediation layer (the "brains")

**Goal:** go beyond "restart the failed thing" — detect an application-level problem (elevated error rate) and have an automated action respond intelligently, not just relaunch a resource.

### 3.1 Create a CloudWatch Alarm on ALB errors
1. CloudWatch Console → Alarms → Create alarm
2. Metric: Namespace `AWS/ApplicationELB`, metric name `HTTPCode_Target_5XX_Count`, dimension = your ALB
3. Statistic: Sum, Period: 1 minute
4. Threshold: e.g., `>= 5` for 1 evaluation period
5. Name: `selfheal-5xx-alarm`

### 3.2 Create an SNS topic for notifications
1. SNS Console → Create topic → Standard → `selfheal-notifications`
2. Subscribe your email so you see every remediation action fire

### 3.3 Create the remediation Lambda
1. Lambda Console → Create function → `selfheal-remediation`
2. Runtime: Python 3.12 or Node 20
3. Give the execution role permissions for whichever action you choose to demonstrate (pick one to start, add more later):
   - **Option A — restart via SSM Automation:** `ssm:StartAutomationExecution` on the built-in `AWS-RestartEC2Instance` document (targets the unhealthy instance by tag)
   - **Option B — force a fresh ECS deployment:** `ecs:UpdateService` with `--force-new-deployment`
   - **Option C — toggle a feature flag:** `ssm:PutParameter` to flip a Parameter Store value your app reads, disabling a broken dependency instead of restarting anything
4. Also grant `sns:Publish` so the Lambda can notify on completion

Sample Lambda logic (Option B, simplest to demo):
```python
import boto3

def handler(event, context):
    ecs = boto3.client('ecs')
    ecs.update_service(
        cluster='selfheal-cluster',
        service='selfheal-service',
        forceNewDeployment=True
    )
    sns = boto3.client('sns')
    sns.publish(
        TopicArn='arn:aws:sns:<region>:<account-id>:selfheal-notifications',
        Message='Remediation triggered: forced new ECS deployment due to elevated 5xx rate.'
    )
```

### 3.4 Wire the alarm to the Lambda via EventBridge
1. EventBridge Console → Rules → Create rule
2. Event source: CloudWatch Alarm State Change
3. Specific alarm: `selfheal-5xx-alarm`, state = ALARM
4. Target: your `selfheal-remediation` Lambda

### 3.5 Test it manually
- Hit the `/error` path repeatedly (or a script/loop) to push the 5xx count over your threshold
- Watch the alarm go to ALARM state → EventBridge fires → Lambda runs → SNS email arrives → ECS shows a new deployment rolling out

This is the closed loop that makes the POC more than a tutorial: **monitor (CloudWatch) → analyze (alarm threshold) → decide and act (Lambda + your chosen remediation) → notify (SNS)**.

## Phase 4: Prove it with fault injection (AWS FIS)

**Goal:** stop manually breaking things and instead use a repeatable, timestamped fault injection so you can measure detection and recovery time.

### 4.1 Create an IAM role for FIS
- FIS needs a service role with permission to terminate EC2 instances / stop ECS tasks. Use the AWS-managed policy path in the FIS console when creating your first experiment template — it will prompt you to create this role.

### 4.2 Create an experiment template for EC2
1. FIS Console → Experiment templates → Create
2. Action: `aws:ec2:terminate-instances`
3. Target: EC2 instances tagged with your ASG's tag (e.g., `aws:autoscaling:groupName = selfheal-asg`), selection mode: 1 instance

### 4.3 Create a second experiment template for ECS
1. Action: `aws:ecs:stop-task`
2. Target: tasks in `selfheal-cluster` / `selfheal-service`

### 4.4 Run the experiments
- Start each experiment from the FIS console and note the start timestamp
- Cross-reference with CloudWatch (ASG activity history / ECS service events) to record how long detection and recovery took

## Phase 5: Observability dashboard

1. CloudWatch Console → Dashboards → Create dashboard `selfheal-dashboard`
2. Add widgets for:
   - ASG "GroupInServiceInstances" over time
   - ECS "RunningTaskCount" vs desired count
   - ALB `HTTPCode_Target_5XX_Count`
   - Alarm state history for `selfheal-5xx-alarm`
3. Run an FIS experiment and screenshot the dashboard showing the dip-and-recovery — this is your best visual for the blog post.

## Phase 6: Cleanup (avoid ongoing charges)

Delete in this order to avoid dependency errors:
1. FIS experiment templates
2. ECS service (scale to 0, then delete) → task definition (deregister) → cluster
3. ECR repository
4. Auto Scaling Group (this terminates its EC2 instances) → launch template
5. ALB → target groups
6. CloudWatch alarms and dashboard
7. EventBridge rule
8. Lambda function and its IAM role
9. SNS topic
